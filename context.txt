### How to reproduce the Colab env

1. Open `colab/001_setup.ipynb` in Colab.
2. Run all cells. The notebook will:
   - Mount Drive under `/content/drive`.
   - Install the approved dependency stack via `setup.install_approved_packages()`.
   - Verify Colab's torch is intact (CUDA available, cuDNN available).
   - Create `MyDrive/fall_detection/{datasets,artifacts,checkpoints,metrics,logs}`.
   - Write a literal `pip freeze` snapshot to `artifacts/pip_freeze.txt`.
   - Write a JSON run log (timestamp UTC, GPU, VRAM, CUDA, Python, Drive paths) to `logs/setup_run_log.json`.

### How to run the manifest validator

```bash
python -m unittest tests.test_manifest_validator -v
```

## Important decisions made in this step

1. **Clip-level role locking.** Every clip carries exactly one role
   (`debug`, `train`, `validate`, `frozen_unseen_test`). Datasets may
   legitimately span multiple roles (Le2i and GMDCSA-24 are documented as
   cross-listed), but each individual clip is unique and single-roled.
   This is enforced by:
   - the dataclass `role: ClipRole` field,
   - the duplicate-`clip_id` check in the validator,
   - the train-vs-validate disjointness check.

2. **Colab-compatible dependency stack.** The approved stack is curated in
   one place (`colab/setup.py:APPROVED_PIP_PACKAGES`) and mirrored in
   `requirements.txt` as lower bounds. Any new dependency must be added
   to both.

3. **No reinstall of Colab torch / torchvision.** The setup explicitly
   omits torch and torchvision from the pip install. Reinstalling them
   from PyPI on Colab risks pulling a CPU-only wheel that breaks every
   GPU model. Colab's pre-built versions are CUDA-matched to the runtime.
   A sanity check (CUDA + cuDNN availability) runs in the notebook after
   install so any silent breakage is caught immediately.

4. **Frozen unseen-test hard wall.** Datasets `omnifall`, `caucafall`,
   `mcfd`, `fallvision` are enumerated in
   `data/manifests/__init__.py:FROZEN_VAULT_DATASETS`. The validator
   rejects:
   - any clip from a vault dataset assigned a non-`frozen_unseen_test` role,
   - any clip assigned `frozen_unseen_test` from a non-vault dataset
     (the role is reserved for the vault).
   This is the strongest leakage guarantee in the project; breaking it
   invalidates every cross-dataset generalisation number we report.

5. **Real `pip freeze` is the env lock.** `colab/setup.py:capture_pip_freeze`
   writes a literal `pip freeze` to `artifacts/pip_freeze.txt`. This
   captures any drift in Colab's base image into version control rather
   than relying on a hand-maintained pin list. `requirements.txt` carries
   lower bounds for human readability; `pip_freeze.txt` is authoritative.

6. **TrackEval is not in the default install.** It has no maintained pip
   wheel. The install command (`pip install git+https://github.com/...`)
   lives in `colab/setup.py:TRACKEVAL_INSTALL_NOTE` and must be run
   manually when MOT metrics from TrackEval are needed.

7. **Validator rules are independently testable.** Each validation rule
   lives in its own private function (`_check_*`). The test suite exercises
   each one in isolation. Adding a new rule is one function + one test.

## Open issues / not yet addressed

- **Real datasets are not yet acquired.** The `sample_manifest.yaml` is
  placeholder. Real clip IDs, `source_path`s, and provenance will land
  when issue 002/003 begin pulling data.
- **Drive layout has no contents yet.** `MyDrive/fall_detection/` will be
  empty until the notebook is first run in a real Colab session.
- **GPU class is not pinned.** The setup captures whatever GPU Colab
  hands the session. The `~16 GB T4` baseline in the PRD is a target, not
  a guarantee — if Colab assigns a smaller GPU (or no GPU at all), the
  downstream pipelines must degrade gracefully.
- **No subject-level leakage check yet.** `subject_id` is captured per
  clip (so the same person in train + validate can be detected later) but
  the validator does not enforce subject disjointness across train /
  validate. Add this once real data is loaded and subject IDs are
  populated.
- **TrackEval install is manual.** No automation; documented in
  `TRACKEVAL_INSTALL_NOTE`.
- **WebDataset shard format not yet chosen.** Issue 003 (clip generator)
  will pick the actual shard naming / tar format. Setup just keeps the
  dependency available.

## Assumptions

- **Python 3.10+ on Colab.** Colab's current image ships 3.10 / 3.11.
  All approved dependencies are compatible.
- **PyYAML is available.** It is in the approved stack and required by the
  manifest loader. The loader surfaces a clear ImportError if missing.
- **Drive mount succeeds.** If running outside Colab, the `DriveLayout`
  falls back to a local path via `FALL_DETECTION_DRIVE_ROOT`.
- **`nvidia-smi` is on PATH inside the Colab runtime.** If not (e.g. CPU
  runtime), the run log records `gpu_name=null` rather than crashing.
- **The dataset slugs in `IN_SCOPE_DATASETS` / `FROZEN_VAULT_DATASETS`
  match the PRD.** Confirmed against `issues/prd.md` Phase I decisions.
- **The `_DASH` convention for dataset slugs is stable.** Source files
  in `data/manifests/sample_manifest.yaml` use lowercase + underscores
  (`up_fall`, `caucafall`) for readability; the loader normalises to
  lowercase so `URFD`, `urfd`, `UrFd` all parse identically.
