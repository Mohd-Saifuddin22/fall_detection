## Critical gotchas for Issue 006 and beyond

### Crop frame_index is 0-based; URFD CSV is 1-based

**This is the single most important gotcha.** Source:
`perception/frames.py` enumerates frames 0-based; the tracker
inherits that index. The crop metadata sidecar persists
`"frame_index": <0-based-absolute>` and the `TrackWindow`
carries the same 0-based values.

The URFD CSV is 1-based: `frame_number` starts at 1.

When joining crop windows to CSV labels, pass
`frame_index_offset=1` to `label_window(...)`. The
`frame_index_offset` parameter exists exactly for this
contract; it is documented on the function signature.

**Issue 006 depends on this.** A wrong offset turns a
near-perfect classifier into a random one with no obvious
failure mode in unit tests.

### CSV coverage is uneven — handle sparse labels

**Fall clips are dense / contiguous.** `fall-01` has
labels 1..160, all present, all in the CSV.

**ADL clips are sparse.** `adl-01` has labels 6..150 with
**frame 7 missing** (an annotator skipped it). The RGB is
dense (1..160) but the CSV is not. Some RGB frames have
no corresponding CSV label row.

`label_window(...)` SKIPS missing frames. A window with
zero available labels returns:
- **`WINDOW_LABEL_UNLABELED`** for fall clips (so Issue 006
  can deterministically drop the example).
- **`(no_fall, False)`** for ADL clips (zero labels is still
  no_fall by source-clip type).

**Do not loosen `gap_tolerance` preemptively.** Only loosen
later if Issue 006 eval proves data starvation.

### CSV label meanings

| Value | Meaning |
| --- | --- |
| `-1` | upright (no fall in progress) |
| `0` | falling / transition |
| `1` | lying on the ground |

### Default window rule (Issue 005 default — pluggable)

| Clip type | Rule |
| --- | --- |
| Fall | `fall` if any available label is `0` or `1`; else `no_fall` |
| ADL | always `no_fall`; sets `is_confuser=True` if any available label is `1` |

Rule is pluggable. Issue 006 can subclass `WindowLabelingRule`
and pass it to `label_window(..., labeling_rule=my_rule)`.

### Zip on-disk shape

- `fall-NN-cam0-rgb.zip` (or `adl-NN-cam0-rgb.zip`) extracts to
  a **double-nested** directory:
  `<staged>/<seq>-cam0-rgb/<seq>-cam0-rgb/*.png`.
- Frame files are `<seq>-cam0-rgb-NNN.png` (1-based, 3-digit
  zero-padded, e.g. `fall-01-cam0-rgb-001.png`).
- `clip_id` for `label_window(...)` is the manifest id
  `urfd-debug-fall-NN-cam0-rgb` (the folder name, lowercased
  for `build_clip_id`).

### Clean rebuild rule (when state is wrong)

Symptom of stale Kaggle data: 21 clips with `(1)`-suffixed
ids. Steps:

1. Restart runtime (clear `/content/fall_local`).
2. Delete Drive `artifacts/crops/`.
3. Delete Drive `artifacts/perception/`.
4. `FORCE_RECOMPUTE = True` in the §1a config cell.
5. Run All.

A fresh full run downloads ~4 GB of university source.

### LOCAL data mode

- Frames are under `/content/fall_local` (wiped on restart).
- Crops and CSVs persist on Drive — they survive `Run All`.
- `frame_index_offset` is the only way to align 0-based
  perception/crop frame_index with 1-based CSV `frame_number`.

### API contracts introduced

1. **Data contracts** (`evaluation.contracts`)

   - `SliceKey(tag: str, value: str)` — hashable label for one slice.
     Validates non-empty fields. `label()` returns `"tag=value"`.
     `to_dict` / `from_dict` for JSON round-trip.
   - `SliceTags(lighting, occlusion, multi_person, action_confuser)` —
     mirrors the optional manifest slice fields. `from_clip(record)`
     builds one from a `ClipRecord`. `keys()` materialises into
     `SliceKey`s; `multi_person=True` renders as `"true"` (string
     for JSON safety).
   - `ClipPrediction(clip_id, score, model_id, dataset, role, slice_tags=None)` —
     one model's output for one clip. `score ∈ ℝ` (range not enforced).
   - `ClipLabel(clip_id, label, dataset, role, source_path, slice_tags=None)` —
     ground truth plus manifest metadata.
   - `EventPredictionStream(clip_id, frame_scores, model_id, clip_start_frame=0, clip_end_frame=0)` —
     ordered `(frame_index, score)` tuples; sparse and dense both
     legal.
   - `EventGroundTruthWindow(clip_id, start_frame, end_frame, label)` —
     inclusive `[start, end]`; `start==end` is a legal single-frame
     window; rejected if `start > end`.
   - `MetricResult(name, value, slice_key=None, higher_is_better=True, notes=None)` —
     value is `float | int | NotAvailable`. `is_available()` checks
     the shape; `numeric_value()` raises if the metric is unavailable.

   All contracts are `frozen=True` dataclasses (mutation blocked).

2. **`NotAvailable(reason, metric_name=None)`** (`evaluation.not_available`)

   - Distinct from numeric 0.0 — a missed metric is not the same
     signal as `the metric read 0.0`.
   - Always falsy (`bool(NotAvailable(...)) is False`) so a guard
     clause `if not metric.value: ...` correctly treats it as
     "skip this metric".
   - Carries a non-empty reason string; carries an optional
     `metric_name` for self-describing aggregation.
   - JSON shape: `{"__not_available__": True, "reason": "...", "metric_name"?: "..."}`.
   - `from_dict` / `is_not_available_marker` / `to_dict` for round-trip.
   - Re-exported as `NOT_AVAILABLE_JSON_KEY = "__not_available__"` so a
     third-party consumer can recognise the marker shape without
     importing the type.

3. **`ExecutionContext` enum + frozen-tier read guard**
   (`evaluation.execution_context`)

   - `ExecutionContext` values: `TRAINING` / `TUNING` / `EVALUATION` /
     `FINAL_JUDGEMENT` / `UNKNOWN`.
   - `FROZEN_ALLOWED_CONTEXTS = {EVALUATION, FINAL_JUDGEMENT}`.
     Training / tuning / unknown **never** receive
     `ClipRole.FROZEN_UNSEEN_TEST` records.
   - `coerce_execution_context(value)` — tolerant of `None`, strings
     (case-insensitive), other enum values, and arbitrary objects;
     unknown values become `UNKNOWN` (fail closed — a default-`None`
     kwarg never grants privileged access).
   - `select_clips_for_context(manifest, context)` — returns the
     manifest minus frozen for non-allowed contexts, full manifest
     for allowed contexts. Always returns a fresh list.
   - `get_frozen_clips(manifest, context)` — explicit frozen accessor
     that **raises `FrozenAccessError`** for non-allowed contexts so a
     training loop that accidentally calls a frozen-aware function
     fails loud instead of silently receiving the non-frozen subset.
   - `enforce_no_frozen_in_iterable(clips, context)` — boundary check
     for hand-built clip lists that bypass `select_clips_for_context`.
   - `frozen_clips_present(manifest)` — diagnostic helper for
     callers that want to assert vault presence before requesting.

   Reuses (does not duplicate) `ClipRole`, `ClipRole.FROZEN_UNSEEN_TEST`,
   `Manifest.by_role()`, and the existing `validate_manifest` vault
   isolation rule.

4. **Result-persistence stub** (`evaluation.result_persistence`)

   - `RESULT_PAYLOAD_FORMAT_VERSION = "1.0"`. Bump this when the
     persisted schema changes in a way that's not transparently
     loadable by older code.
   - `EvalRunMetadata(run_id, model_id, created_at, context, notes=None)`
     — provenance for one run. `make_default_metadata()` stamps the
     current UTC ISO-8601 timestamp.
   - `MetricResultPayload(format_version, metadata, metrics)` —
     `to_dict` / `from_dict` for round-trip; `from_dict` is strict on
     shape and `format_version` so a schema bump fails loud instead
     of silently losing fields.
   - `MetricResultStore(metrics_root: Path | str)` — writes
     `<root>/<run_id>/results.json` + `<root>/<run_id>/summary.txt`.
     `run_id` is validated against path separators so it cannot
     escape its directory.
   - `encode_value(value, metric_name=None)` — JSON-friendly
     renderer for a single metric value: numeric passes through as
     `float`; `NotAvailable` renders as the marker dict, with the
     `metric_name` kwarg stamping the marker when it's not already
     set (so a payload consumer that lost the surrounding record
     ordering can still pair marker → metric name).
   - The store accepts a plain `Path` or string — never a hardcoded
     Drive path. The integration test confirms it works against a
     `DataLayout.metrics`-style root (`<root>/metrics`).

### Decisions made

- **`NotAvailable` is a dataclass, not `None` or `NaN`.** A bare
  `None` is too generic; `float("nan")` is hard to grep for and
  silently aggregates wrong. A dedicated type makes the semantics
  explicit at the value layer.
- **The frozen guard fails closed by default.** `UNKNOWN` is
  treated as a non-allowed context, and `coerce_execution_context`
  maps any non-enum / non-string value to `UNKNOWN`. Adding a new
  privileged context is a one-line change in
  `FROZEN_ALLOWED_CONTEXTS` and should be deliberate.
- **`get_frozen_clips` raises, `select_clips_for_context` filters.**
  Two entry points on purpose: the filter is for callers who want
  "whatever's appropriate for this context"; the raise is for
  callers who specifically asked for frozen and would notice if
  they accidentally got an empty list. Code that wants a frozen-
  tier "yes I really meant it" assertion uses the raise variant.
- **The guard reuses, does not replace, the validator.** The
  existing `_check_frozen_vault_isolation` rule catches manifests
  where a vault dataset leaked into train/validate. The guard is a
  second layer against accidental context misuse at READ time.
  Step 1 tests prove both layers coexist without one shadowing the
  other.
- **Result persistence is format-versioned at the payload root.**
  Schema bumps can be detected immediately on load; the loader
  refuses unknown `format_version` rather than guessing.
- **`run_id` is a filename segment, not a path.** Path separators
  are rejected in `MetricResultStore.save()` so a `run_id` cannot
  escape its directory.
- **Stub, not implementation.** Step 1 deliberately ships only the
  container. No classifier produces metric results yet, so there is
  no `evaluation.metrics.classification` or `evaluation.metrics.event`
  module. Step 2+ adds those.

### Assumptions

- **Step 1 ships with no real trained models.** Therefore no
  classification metrics are implemented, no aggregate / sliced
  numbers are produced, and the persistence stub is exercised
  with hand-built fixtures only.
- **The active layout concept from Issue 002 carry-over still
  applies.** `MetricResultStore` accepts a path; callers are
  responsible for passing `layout.metrics` (or equivalent). No
  Drive paths are baked into the eval layer.
- **`Manifest.by_role()` is stable.** The guard depends on it
  returning an iterable of `ClipRecord` rows, ordered as the
  manifest has them. If the loader is extended (e.g. to add an
  indexing data structure), the guard should be revisited.
- **`NotAvailable` is intentional silence, not an error.** Callers
  that want a hard error when a metric is missing should check
  `result.is_available()` and raise; the eval harness itself does
  not raise on `NotAvailable`.

## Issue 004 Step 1 — governance correction (final-judgement only)

The initial Step 1 draft allowed `EVALUATION` to read frozen records
("operator may want to spot-check frozen behaviour without it being on
the critical path"). The required governance rule is stricter: **the
vault opens only on final judgement**, never on a routine eval call.
This section documents the correction.

### What changed

- `evaluation/execution_context.py`
  - `FROZEN_ALLOWED_CONTEXTS` is now
    `frozenset({ExecutionContext.FINAL_JUDGEMENT})` — a single-entry
    allow-list. `EVALUATION` is removed.
  - The module / enum docstrings now state, explicitly: *"the vault
    opens ONLY for ``FINAL_JUDGEMENT``"*. The prior "operator
    spot-check" rationale is removed.
  - `is_frozen_allowed(...)` still returns `True` for exactly
    `ExecutionContext.FINAL_JUDGEMENT` and `False` for everything
    else.
  - `FrozenAccessError` is now produced for `EVALUATION` (in both
    enum and string form) at both read entry points:
    `select_clips_for_context` and `get_frozen_clips`. The error
    message's allow-list slice now reads `Only ['final_judgement']`.

- `tests/test_eval_frozen_guard.py`
  - `test_only_final_judgement_can_see_frozen` asserts the new
    single-entry allow-list.
  - `test_evaluation_is_not_in_the_allow_list` — belt-and-braces
    assertion: `EVALUATION` must NOT be in the allow-list, period.
    A future regression that re-adds it would fail this test
    loud.
  - `test_evaluation_context_excludes_frozen_clips` +
    `test_evaluation_string_form_also_excludes_frozen` replace
    the prior *includes* cases.
  - `test_evaluation_now_raises_frozen_access_error` +

### Why this is a strict widening, not a relaxation

Any code path that previously read frozen under `EVALUATION` now

### Optional cleanup landed alongside the governance fix

- **`NotAvailable.__bool__` comment fixed.** The prior comment
  parsed as "if not_available: ... is True", which contradicted the
  implementation. Rewritten to enumerate the four falsy / truthy
  cases unambiguously: `bool(na)` is False; `if na:` is False;
  `if not na:` is True; `if na.value:` is False when the value is
  the marker. The docstring on `__bool__` is now self-consistent.

- **Result-store overwrite behaviour hardened and documented.**

    - A `run_id` is the result's identity on disk; a silent

    - `test_overwrite_default_refuses_to_replace_existing_run` —
      default behaviour surfaces `FileExistsError`.
    - `test_overwrite_true_replaces_existing_run` — explicit
      opt-in replaces the payload.
    - `test_overwrite_only_clears_named_payload_files` — sibling
      files placed in `run_dir` survive a re-save.

  The store still has no `delete()` method: a misclick that drops
  results is worse than one that requires a few lines of
  `shutil.rmtree(store.run_dir("..."))` to undo.

### Metric API contract

```
from evaluation import compute_classification_metrics

reports: list[SliceMetricReport] = compute_classification_metrics(
    predictions: Iterable[ClipPrediction],
    labels: Iterable[ClipLabel],
    *,
    threshold: float = 0.5,
    slice_tags: Sequence[str] = ("dataset", "lighting", "occlusion",
                                 "multi_person", "action_confuser"),
) -> list[SliceMetricReport]
```

The returned list starts with the aggregate
(`report.slice_key is None`), followed by one report per emitted
`(tag, value)` slice. Slice ordering is deterministic: by-tag
alphabetical, then by-value alphabetical.

Each :class:`SliceMetricReport` carries:

- `slice_key: SliceKey | None` — `None` for the aggregate.
- `support: SupportCounts` — typed `n_positive` / `n_negative` /
  `total`.
- `confusion_matrix: ConfusionMatrix` — typed `tn` / `fp` / `fn` /
  `tp` (sklearn convention with `labels=[0, 1]`).
- `metrics: tuple[MetricResult, ...]` — per-slice rows for the
  seven primary metric names: `accuracy`, `precision`, `recall`,
  `specificity`, `f1`, `auc_roc`, `auprc`. Each value is a real
  `float` or `NotAvailable` with reason.

`report.metric_results()` returns the flat row list for
persistence: `metrics` rows + four `tn` / `fp` / `fn` / `tp` rows +
three `n_positive` / `n_negative` / `total` rows. The
:class:`MetricResultStore` accepts this list unchanged.

### Threshold behaviour

- **Default 0.5** (`DEFAULT_THRESHOLD`). Count-based metrics
  (accuracy, precision, recall, specificity, F1) use
  `score >= threshold` as the predicted "fall" hard label.
- **AUC-ROC and AUPRC ignore the threshold.** They use the raw
  score distribution and are threshold-independent by
  construction. Two runs with the same data but different
  thresholds must produce identical AUC-ROC / AUPRC values
  (`test_auc_roc_uses_raw_scores_threshold_independent`,
  `test_auprc_uses_raw_scores_threshold_independent`).

### NotAvailable behaviour (honest, never fabricated)

| Condition | Result |
|---|---|
| Empty `predictions` AND empty `labels` (no pairs) | empty list returned; no spurious aggregate. |
| Per-slice eval with 0 pairs in the slice | every metric `NotAvailable(reason="empty slice")`; typed `support` and `confusion_matrix` still emitted as zero-valued. |
| AUC-ROC on a slice with only one class | `NotAvailable(reason="only one class present in slice", metric_name="auc_roc")`. Detected explicitly via `len(unique_classes) < 2` rather than relying on sklearn's exception (sklearn's behaviour has drifted across versions — newer releases emit warnings + return `0.0` instead of raising). |
| AUPRC on a slice with only one class | `NotAvailable(reason="only one class present in slice", metric_name="auprc")`. |
| Precision on a slice where the model never predicts positive (`tp + fp == 0`) | `NotAvailable(reason="no predicted positives in slice", metric_name="precision")` — the denominator is literally undefined. Fixed in the Step 2 cleanup. |
| Recall on a slice with no actual positives (`tp + fn == 0`) | `NotAvailable(reason="no actual positives in slice", metric_name="recall")`. Fixed in the Step 2 cleanup. |
| Specificity on a slice with `n_negative = 0` | `NotAvailable(reason="no negatives in slice")` — fabricating 1.0 here would silently claim "no false positives" without any negatives to test against. |
| F1 on a slice where precision OR recall is undefined | `NotAvailable(reason="precision or recall undefined", metric_name="f1")`. Fixed in the Step 2 cleanup. |
| Count-based metrics on a one-class slice (e.g. accuracy when only `FALL` clips are present) | numeric, computed from sklearn as normal — the metric IS defined when the denominator is non-zero. |

Sklearn parity is asserted explicitly — `test_auc_roc_matches_sklearn_directly` and `test_auprc_matches_sklearn_directly` compute the same numbers with `sklearn.metrics.roc_auc_score` / `sklearn.metrics.average_precision_score` and compare to within 12 places. No hand-rolled TP/FP/TN/FN math drives any output.

### Pairing + failure modes

`compute_classification_metrics` pairs predictions to labels by `clip_id`. The contract enforces one-prediction-per-clip and one-label-per-clip on BOTH sides:

- Duplicate clip_id in predictions → `ValueError`.
- Duplicate clip_id in labels → `ValueError`.
- Prediction with no matching label → `ValueError`.
- Label with no matching prediction → `ValueError`.

Empty input (`predictions=[]` and `labels=[]`) returns an empty list — the caller genuinely has no data, so no aggregate / slice is fabricated.

### Slice semantics

| Input | Behaviour |
|---|---|
| Clip with `slice_tags=None` or `SliceTags(lighting=None, ...)` | appears in the aggregate only; contributes to no per-tag slice. |
| Clip with `lighting="daylight"` but no `action_confuser` value | appears in `lighting=daylight` slice AND in the aggregate; does not appear in any `action_confuser=...` slice. |
| Per-tag defaults | `("dataset", "lighting", "occlusion", "multi_person", "action_confuser")` — the PRD-mandated axes. Configurable per call. |
| `multi_person` rendering | the manifest field is a `bool`; `SliceTags.keys()` renders it as `"true"` / `"false"` strings for stable JSON round-trip and per-slice grouping. |

Per-slice support counts are correct by construction: a clip with `lighting="daylight"` enters exactly one `lighting=...` slice, no more. Multiple clip totals across slices do NOT sum to the aggregate because a single clip can enter multiple slices (one per tag it has values for).

### Recall / AUPRC priority metrics

Per PRD user story 38 ("false negatives treated as the most costly error") and user story 22 (AUPRC + recall must be honest), `recall` and `auprc` are emitted alongside the other count metrics — not aggregated or hidden. Their threshold behaviour and per-slice parity is the same as the count metrics they pair with.

## Issue 004 Step 2 — metric-honesty cleanup

Step 2's first pass on classification metrics relied on sklearn's
`zero_division=0` knob for precision / recall / F1 — silently
fabricating `0.0` when the denominator was undefined. The cleanup
removes that dependency entirely; degenerate-slice metric values are
now reported as `NotAvailable` with a precise reason rather than as
a fake `0.0`.

### What changed

- `evaluation/metrics/classification.py`
  - Module docstring adds the full degenerate-slice contract as a
    table; same table is mirrored in `context.txt`.
  - `compute_classification_metrics` docstring adds a
    "Degenerate-slice semantics" bullet.
  - The dispatch helper `_sklearn_compute` and the placeholder
    wrapper `_emit_metric_pair` are gone. A new single source of
    truth, `_emit_metrics_for_slice`, builds all seven primary
    `MetricResult` rows from a single computed confusion matrix +
    raw scores.
  - **`_compute_one_slice_report` is deleted** (it was dead — the
    actual entry path went through an inner `_report` function).
  - `_compute_auc_roc` / `_compute_auprc` return annotations are
    tightened to `float | NotAvailable` to reflect the two-shape
    return type.
  - `zero_division=0` is no longer passed to `precision_score`,
    `recall_score`, or `f1_score`. We never call sklearn on a
    undefined denominator; instead we gate the call on
    `tp + fp > 0` / `tp + fn > 0` and surface `NotAvailable` when
    the gate fails.
  - `_thresholded_pairs` is hoisted out of the inner closure so
    it can be shared by the empty / non-empty report paths; the
    non-empty report path now delegates to
    `_emit_metrics_for_slice`.
  - `_empty_report` is hoisted out as a named helper to keep the
    `_report` function readable.

- `tests/test_eval_classification.py`
  - Three new test classes (`AllNegativeSliceTests`,
    `AllPositiveSliceTests`, `ActionConfuserSittingTests`) cover
    the contract:

        - `test_all_no_fall_no_predicted_positives_is_fully_undefined`
          — when the model never predicts positive AND the slice
          has no actual positives, every degenerate metric becomes
          NotAvailable with the precise reason
          (`"no predicted positives in slice"`,
          `"no actual positives in slice"`, etc.).

        - `test_all_no_fall_but_some_predicted_positives_has_numeric_precision`
          — when some predictions cross the threshold, precision
          becomes well-defined (and is `0.0` because `tp=0`),
          but recall / F1 stay NotAvailable.

        - `test_all_fall_slice_is_numeric_for_recall_only` —
          symmetric case: recall numeric, specificity
          NotAvailable.

        - `test_all_fall_with_no_predicted_positives_is_fully_undefined`
          — pathological case combining both degeneracies.

        - `test_action_confuser_sitting_all_no_fall_slice` —
          PRD-flavoured case: `action_confuser=sitting` with all
          `no_fall` labels. The slice's false-positive count (1)
          is visible in both `cm.fp` and the persisted
          `MetricResult(name="fp")` row.

### Why this is correct, not just stricter

A degenerate slice has a denominator of zero. The mathematical
value is **undefined**, not zero. Reporting `0.0` for "precision
on a slice where the model never predicts positive" silently reads
as "the model is wrong about every positive it predicted" — but
there *are* no predicted positives, so the statement is vacuous.
Falsifying `0.0` in this corner poisons any downstream selector /
aggregator that compares against `> 0` thresholds or averages
across heterogeneous slices; an unavailable metric must surface as
unavailable so the aggregator can decide whether to skip the slice,
skip the metric, or skip the model entirely.

Conversely, a numeric `0.0` reading on a *defined* denominator
(e.g. `tp=0, fp>0` so `precision = 0/1 = 0.0`) IS a real result —
the model got every predicted positive wrong. The contract
distinguishes those two cases precisely: `tp + fp == 0` →
`NotAvailable`; `tp + fp > 0` → numeric (sklearn). The same rule
applies to recall, F1, and specificity.

### Self-test API contract

```python
from evaluation import (
    SelfTestConfig,
    SelfTestResult,
    SyntheticClassificationSet,
    build_synthetic_baseline,
    run_classification_self_test,
    shuffle_synthetic_labels,
)

cfg = SelfTestConfig()                # deterministic defaults
result: SelfTestResult = run_classification_self_test(cfg)
# result.baseline_auc_roc           — float
# result.shuffled_auc_roc_values    — tuple[float]
# result.shuffled_auc_roc_mean      — float
# result.degradation_margin         — float (baseline - shuffled_mean)
# result.degraded                   — bool (pass-criteria all hold)
# result.passed                     — bool (same as degraded; field
#                                     exists for explicit semantics)
# result.reason                     — str ("all checks passed" or a
#                                     semicolon-joined list of
#                                     which checks fired)
# result.n_shuffles, margin,
#   config_seed_base, healthy_auc_range,
#   degraded_auc_range              — knobs echoed for traceability
```

`run_classification_self_test` accepts an optional
``evaluate_metrics`` callable that defaults to the real
`compute_classification_metrics`. Injecting an evaluator that
ignores inputs and reports a constant AUC-ROC is how the test
proves it has teeth.

### Deterministic shuffle policy

- Synthetic builder is seeded with ``config.seed_base`` via an
  explicit ``random.Random(seed)``. No global RNG is consulted.
- Shuffle ``i`` uses seed ``config.seed_base + i + 1`` (deterministic
  and easy to reproduce — `SelfTestConfig.shuffled_seeds()` exposes
  the exact list).
- A single shuffle permutes labels and renames each label's
  ``clip_id`` to the prediction's ``clip_id`` at the shuffled
  index. Renaming ``clip_id`` is what actually corrupts the
  ``(score, label)`` bond — without it, the metric bundle would
  re-pair by ``clip_id`` and the shuffle would be a no-op.
- ``n_shuffles = 8`` (default) keeps the mean non-flaky at the
  default sample size; the test asserts the result is invariant
  across two runs of the same config.
- No floating-point variance: the exact same inputs go through
  the exact same sklearn paths (no shuffling inside
  `compute_classification_metrics`).

### Default knob values

| Knob | Default | Rationale |
|---|---|---|
| `n_samples` | 200 | Enough for stable AUC-ROC; runs in well under a second. |
| `score_class_separation` | 0.6 | Yields a baseline AUC-ROC near 1.0 against the original labels. |
| `positive_rate` | 0.4 | Slight fall imbalance — falls are intrinsically rare in this domain; the exact split keeps both AUC computations well-defined. |
| `n_shuffles` | 8 | Stable mean without being slow. |
| `seed_base` | 12345 | Arbitrary fixed seed. |
| `margin` | 0.30 | Required baseline-vs-shuffled gap. |
| `healthy_auc_range` | `(0.85, 1.001)` | Half-open upper bound — a constant 1.000 reading is acceptable. |
| `degraded_auc_range` | `(0.4, 0.6)` | PRD's "approximately chance" band. |
| `score_noise` | 0.10 | Per-clip score jitter (gaussian sigma). |

### Pass criteria

`passed = True` iff all three checks hold:

1. ``healthy_auc_range[0] <= baseline_auc_roc < healthy_auc_range[1]`` — the
   synthetic baseline is genuinely well-separated.
2. ``degraded_auc_range[0] <= shuffled_mean <= degraded_auc_range[1]`` —
   shuffled labels collapse the AUC toward chance.
3. ``degradation_margin >= margin`` (default 0.30) — the gap between
   baseline and shuffled is large enough that the harness clearly
   noticed the corruption.

Any failed check is appended to ``result.reason`` so a CI log
points at the specific axis that failed.

### Teeth (broken-scorer test)

`tests/test_eval_self_test.py::BrokenScorerTests` injects a
wrapper that runs the real metric bundle but overwrites every
aggregate's AUC-ROC with a constant. For values ``{0.5, 0.7,
0.95, 0.99, 1.0}`` the self-test consistently reports
``passed = False`` and a precise ``reason`` string. That's the
mechanism by which a broken downstream aggregator / scorer
(silent always-great-AUC, corrupted label-pipe, etc.) trips the
self-test rather than passing silently.

### Extension seam (future sentinels — NOT implemented here)

The same ``build synthetic → deterministic shuffle → real harness →
sentinel metric → pass-criteria`` pattern extends to the rest of the
PRD's self-test plan. The module docstring lists:

- **Tracking sentinel:** synthetic ByteTrack output with injected
  ID switches; sentinel metric is IDF1 / MOTA / HOTA on the
  future event-metric bundle.
- **Skeleton sentinel:** synthetic keypoint sequence with
  degraded confidence; sentinel metric is missing-keypoint rate.
- **Post-verification sentinel:** degraded fall score sequence;
  sentinel metric is detection delay.

A future ``run_<component>_self_test(...)`` function in this
module (or a new ``evaluation/self_test/<component>.py``) mirrors
the same shape: ``Synthetic<Component>`` builder + a deterministic
corruption strategy + a sentinel reading + a
``run_<component>_self_test`` orchestrator returning a
``SelfTestResult``.

The four-tier pipeline:

1. **Alert derivation** — pure; either via the bundle
   wrapper (`compute_event_metrics_for_stream`) or by a future
   Pipeline C post-verification engine handing off its alert
   frames to `compute_event_metrics_for_clip` directly.
2. **Match** — `match_alerts_to_events(alerts, events, tolerance=0)`
   returns an `EventMatching` carrying matched alerts / matched
   events / unmatched alerts / unmatched events / per-match
   delays-in-frames. First-match-per-event wins; positive
   tolerance swallows small lead-ins and trailing edges.
3. **Per-clip bundle** — `EventMetricBundle` exposes
   `metric_results()` that flattens everything into
   `MetricResult` rows the Step 1 `MetricResultStore`
   accepts unchanged.
4. **Cross-dataset aggregation** — `aggregate_event_metrics_by_dataset`

### Alert rule defaults

The defaults match the PRD's post-verification starter:

| Knob | Default | Source of truth |
|---|---|---|
| `threshold` | `0.80` | `DEFAULT_ALERT_THRESHOLD` |
| `persistence` | `10` (consecutive frames) | `DEFAULT_ALERT_PERSISTENCE` |
| `tolerance` | `0` (alert frame must lie inside the GT window) | `DEFAULT_EVENT_TOLERANCE_FRAMES` |

`AlertRule` is a frozen dataclass — its fields are validated at

three defaults are exposed as module constants so callers that
do not want to thread an `AlertRule` instance can still refer
to them by name.

The alert fires at the *last* frame of a `persistence`-sized
chunk of consecutive above-threshold scores. After firing,
the run is reset so a sustained high-score region produces
one alert per chunk rather than one per frame.

| Condition | Result |

## Issue 004 Step 4 — event-matching correctness fix

The first pass of Step 4 honoured "first-in-window alert per
event" but counted any alert that didn't credit an unmatched
event as a false alarm. That misclassified redundant in-window
alerts inside an already-matched event. The fix:

### Corrected event-matching contract

1. **False alarm.** An alert is a false alarm iff it falls
   outside every ground-truth event window (including tolerance).
2. **Matched GT event.** A GT event is matched if at least one
   alert falls in its tolerated window.
3. **First-match-per-event wins.** The first in-window alert
   credits the event and sets the detection delay. Subsequent
   alerts in the same window are silently dropped.
4. **Redundant in-window alerts are harmless.** An alert that
   falls in an already-matched event's window is neither an
   additional match nor a false alarm. It does not appear in
   ``unmatched_alerts`` and does not inflate the
   false-alarm count or false-alarms-per-hour.
5. **Unmatched events** are GT events with zero in-window
   alerts.
6. **Spurious alerts** are alerts outside every tolerated GT
   window.

### Corrected precision formula

```text
recall    = matched_event_count / total_gt_events
precision = matched_event_count / (matched_event_count + false_alarm_count)
f1        = 2 · precision · recall / (precision + recall)
```

The numerator is the count of GT events credited — **not** the
count of matched alerts. The denominator counts both credited
events and spurious alerts (false alarms). Redundant in-window
alerts do not enter either side.

### Where the change shows up

- `match_alerts_to_events` (`evaluation/metrics/event.py`) has
  a second pass: when no unmatched event covers the alert,
  walk every event's window and silently drop the alert if any
  covers it. Only outside-every-window alerts become
  `unmatched_alerts`.
- `EventMetricBundle.metric_results()` computes
  `event_precision = matched_events / (matched_events +
  len(unmatched_alerts))`. Previously it was
  `matched_alerts / (matched_alerts + len(unmatched_alerts))`.
- `aggregate_event_metrics_by_dataset` reflects the same
  numerator shift: per-dataset `event_precision` uses the
  count of matched events across the dataset's bundles, not
  the count of matched alerts.
- `false_alarms_per_hour` continues to use
  `len(unmatched_alerts)` — but with the redundant alerts
  filtered out at the matcher, that count is now an honest
  spurious-alert count.

### Files changed

| Path | Change |
|---|---|
| `evaluation/metrics/event.py` | Matcher's second-pass "is this alert inside any window?" check. Precision formula updated to `matched_events / (matched_events + len(unmatched_alerts))`. Aggregator's per-dataset precision mirrors the same numerator. |
| `tests/test_eval_event.py` | New: `test_redundant_in_window_alerts_are_not_false_alarms` and `test_redundant_in_window_does_not_change_per_metric_results` (the brief's example: GT [100, 130], alerts 109/119/129 → false_alarms=0, recall=1.0, precision=1.0, delay=9 frames). ## Issue 004 Step 5 — Le2i ground-truth adapter

Step 5 lands the first dataset-specific annotation parser. The
parser is fully unit-tested against small in-memory text fixtures
that match the verified Le2i annotation format — no Kaggle
download, no real video file, no OpenCV dependency at test time.

### Files created

| Path | Purpose |
|---|---|
| `data/le2i.py` | Annotation parser, per-frame detection container, video/annotation pairing helpers, FPS helper. |
| `tests/test_le2i_parser.py` | 36 unit tests: no-fall fixture, fall fixture, malformed input (raise), all-zero bbox (present=False), missing-file path (returns None), non-contiguous stems, nested same-named subfolder resolution. |

### Le2i annotation contract

```
<fall_start_frame>          # line 1, integer, 1-based
<fall_end_frame>            # line 2, integer, 1-based
<frame_index>, <flag>, <x1>, <y1>, <x2>, <y2>   # line 3+
...
```

- **Lines 1 + 2.** Inclusive 1-based frame indices of the fall
  window. ``0, 0`` is the **no-fall sentinel** and produces no
  event window. Mixed ``(0, X)`` / ``(X, 0)``, inverted
  (start > end), and zero-indexed windows all raise
  `ValueError` — fail loud.
- **Line 3+.** Per-frame detection row. ``frame_index`` is
  1-based; ``flag`` is preserved verbatim (the parser does
  not interpret it); ``x1, y1, x2, y2`` are pixel-corner
  coordinates.
- **All-zero box.** ``0, 0, 0, 0`` is the absent-person marker;
  the row is preserved with ``present=False`` (so a
  downstream consumer can skip without losing the row's
  presence) and never becomes a GT box.
- **Malformed non-zero box.** ``x1 >= x2`` or ``y1 >= y2`` is
  rejected; when ``frame_size`` is supplied, the box must also
  fit inside the frame (`0 <= x1 < x2 <= width`, etc.).

### No-fall sentinel behaviour

`parse_le2i_annotation_text` parses a ``0 0`` first-two-lines
fixture into a `Le2iAnnotation` whose `fall_window` is the
empty tuple, `has_fall` is `False`, and `event_window()` is
`None`. No event is created — the metric bundle treats a
no-fall clip as "no event GT available" via `NotAvailable`,
never as a fabricated 0/0 → 0.0 reading.

### Missing-GT behaviour

`parse_le2i_annotation(path)` returns `None` when the file
does not exist; the helper does **not** raise. `collect_le2i_clips`
records `annotation=None` on the resulting `Le2iVideoPair`,
so the clip is preserved in the inventory but the metric
bundle can refuse to compute event-level metrics for it. FPS
still surfaces (with fallback if configured) so
false-alarms-per-hour + detection-delay-in-seconds remain
computable on the missing-GT clip without requiring GT.

The brief's "Office and Lecture-room may have videos but no GT"
case maps to this path exactly — those clips are walked, the
absence is recorded honestly, downstream consumers can branch
on `annotation is None`.

### FPS fallback behaviour

`read_le2i_fps(video_path, *, fallback_fps=None)` reads the
FPS via OpenCV (lazy-imported so the parser is usable in
environments where OpenCV is not yet wired). Behaviour:

| OpenCV status | `fallback_fps` provided? | Returned value |
|---|---|---|
| Available, FPS > 0 | any | the float FPS |
| Available, but file unreadable / FPS == 0 | yes | `fallback_fps` (float) |
| Available, but file unreadable / FPS == 0 | no | `NotAvailable("cannot read FPS from .avi")` |
| OpenCV import fails | yes | `fallback_fps` (float) |
| OpenCV import fails | no | `NotAvailable("opencv-python not installed")` |

The fallback is **never** silently applied to a path that
succeeds at reading FPS — only failures route through the
fallback. With no fallback, failures are surfaced as
`NotAvailable` rather than fabricated as `25.0`. The metric
bundle's fps-less paths (frames-based delays, false-alarm count)
remain computable in either case.

### Pairing helpers

| Helper | Behaviour |
|---|---|
| `pair_video_with_annotation(video, annotations_root)` | Pair by filename stem; search order is `annotations_root / stem.txt` then `annotations_root / stem / stem.txt` (nested same-name subfolder). Returns the matched `.txt` path or `None`. |
| `collect_le2i_clips(videos_root, *, annotations_root=None, fallback_fps=None, frame_size=None)` | Walk `videos_root` (any depth) for `.avi`/`.mp4` files, pair each with its annotation via `pair_video_with_annotation`, attach the per-pair FPS read. `annotation=None` is preserved; videos in subfolders are picked up. |

Non-contiguous stems work without special-casing: the walk
finds every `.avi` regardless of gaps in numbering, and the
pairer only cares about the stem-to-stem match on each
individual file.

### What Step 5 explicitly does not do

- No training, no model code, no metric computation. The
  parser produces `Le2iAnnotation` / `Le2iVideoPair` records
  and lets downstream modules (cropping, metrics) consume
  them.
- No real Kaggle download. Tests run on inline text fixtures.
- No real video reads. `read_le2i_fps` is exercised only on
  the missing-file and fallback-configured paths at test
  time; the OpenCV-succeeds path is exercised when real
  Le2i videos land on disk.
- No skeleton / pose-adapter (PCK at event level stays in
  `compute_component_metrics` as a `NotAvailable` row).

## Issue 002 decisions and rules

### 1. URFD source and staging

- **Source.** `tanmaydacha/urfd-dataset` on Kaggle. Whitelisted in
  `data/stage_urfd.py:ALLOWED_KAGGLE_SLUG`. Any other slug raises before
  any network call.
- **Destination.** `MyDrive/fall_detection/datasets/urfd/`.
- **Idempotency.** A marker file `.staged_from_kaggle.txt` is written
  on success; re-runs short-circuit. `force=True` overrides for the
  rare case where the staged tree is corrupted.
- **Provenance.** The marker records the slug and the credential source
  ("colab_secrets") — never the credential values themselves.

### 2. Kaggle credential handling rule

- **Read from Colab Secrets at runtime.** `KAGGLE_USERNAME` and
  `KAGGLE_KEY` from `google.colab.userdata`.
- **Never** print, log, return, commit, or write the values anywhere.
  The staging script's only credential-aware code path sets two env
  vars and returns; the values never leave the function.
- **Local-dev fallback.** Outside Colab, the script looks for
  `~/.kaggle/kaggle.json` on disk. We do NOT auto-create it.
- **No fallback to env vars or files outside the two paths above.**
  This keeps the credential surface tiny and auditable.

### 3. Folder-of-frames input contract

The tracker accepts an **ordered** iterable of frame paths:

- The caller (the notebook / orchestrator) is responsible for ordering.
- `FrameFolderReader` enforces numeric sort by trailing integer in the
  filename stem (`frame_0001.png` → 1, `fall-01-cam0_frame_0042.png`
  → 42). Non-numeric frames sort last by filename.
- Out-of-order frames produce silently invalid tracks. The bug is
  invisible until you visualise the annotated output. **Never bypass
  the loader.**

### 4. Numeric frame sorting requirement

- Implemented in `perception/frames.py:discover_frames`.
- Sort key: `(has_number_flag, trailing_integer, filename)`.
- Frames with a trailing integer sort by that integer ascending.
- Frames without one sort to the END (so they don't poison the start
  of the timeline).
- Ties on the integer (rare) are broken by full filename (lexical) for
  determinism.

### 5. YOLO26 + ByteTrack configuration

| Knob | Value | Source of truth |
|---|---|---|
| `model_name` | `yolo26m` | `perception/tracker.py:REQUIRED_MODEL` |
| `tracker_config` | `bytetrack.yaml` | `perception/tracker.py:REQUIRED_TRACKER` |
| `persist` | `True` | (required for ByteTrack ID survival across frames) |
| `classes` | `[0]` (person) | `perception/tracker.py:PERSON_CLASS_ID` |
| `conf` | `0.25` | `TrackerConfig` default |

**Hard rule:** any model name other than `yolo26m` raises
`UnsupportedModelError`. No silent fallback to yolo11 / yolov8 / etc.
Pretrained weights only — no training in this module.

### 6. Metrics limitation due to missing ground truth

URFD ships with raw frames only — no detection or tracking labels.
The report therefore does NOT compute mAP, IDF1, MOTA, or HOTA. They
are recorded in `metric_availability` as:

```python
"map_50":     "n/a (no detection ground truth)"
"map_50_95":  "n/a (no detection ground truth)"
"idf1":       "n/a (no tracking ground truth)"
"mota":       "n/a (no tracking ground truth)"
"hota":       "n/a (no tracking ground truth)"
```

Debug-tier validation is **annotated visual review** + the
track-continuity / fragmentation report. The PRD already calls this
out as the "cheapest killer risk to retire first" — track-through-fall
must survive the fall motion itself.

### 7. Fallback levers (apply only if the baseline fails)

In escalation order, exposed on `TrackerConfig`:

1. `fallback_track_low_thresh` — lower ByteTrack's association threshold
   (e.g. `0.1`) so weak detections still get associated. **Auto-wired**
   via Ultralytics ``cfg={"track_low_thresh": ...}`` — ByteTrack and
   BoT-SORT both honour this key.
2. `fallback_tracker='botsort.yaml'` — switch tracker config to BoT-SORT
   (different association algorithm). **Auto-wired** via
   ``tracker=`` arg.
3. `fallback_end2end=False` — disable the BoT-SORT end-to-end pass.
   **NOT auto-wired.** ``end2end`` is a model/runtime constructor arg of
   the BoT-SORT class, not a tracker config key — it cannot be expressed
   through ``model.track()``'s ``cfg=``. The field is kept on
   ``TrackerConfig`` so the *intent* is recorded in ``run_meta.json``
   (with the ``manual intervention required`` annotation); applying it
   requires a manual code change.

Whichever lever was used is recorded in ``run_meta.json`` as
``fallback_used``. Auto-wired levers appear bare; manual-only levers
appear with the ``(manual intervention required)`` suffix so a human
reviewer can tell at a glance what was actually applied.

Default ``TrackerConfig()`` has every fallback set to ``None`` — the
baseline is what runs by default. **Do not run fallback levers unless
the baseline real URFD result proves weak** (Issue 002 rule).

### 8. URFD manifest shape

- Schema version: `1.1`.
- All clips are `role: debug` (URFD is a debug-tier dataset).
- Labels:
  - `fall-NN-camM` → `fall`
  - `adl-NN-camM` → `no_fall`
- `source_path` is **Drive-relative**: `datasets/urfd/<folder_name>`.
- `notes` carries `sequence=NN; camera=camM; frame-folder (PNGs)`
  for downstream traceability.

| Field | Default | Source of truth |
|---|---|---|

#### Short-track / gap / min-coverage policy

- **Empty track:** skipped with reason `"empty_track"`.
- **Track shorter than `clip_length`:** every candidate window is
  dropped (consecutive missing frames exceed `gap_tolerance`); reason
  `"too_many_gaps"`.
- **Gap inside a window:** filled by carry-forward from the last
  real box. The missing-frame slot carries a placeholder geometry
  so downstream cropping still produces a uniform canvas.
- **Gap run longer than `gap_tolerance`:** the entire window is
  dropped with `"too_many_gaps"`.
- **Tiny boxes (avg area < `min_box_area_px` OR median side <
  `min_box_side_px`):** window dropped with `"insufficient_coverage"`.

Every skip is recorded in the run summary's `skip_reason_counts`.

#### Shard output format

- Format: WebDataset-style `.tar` shards with one image + one JSON
  metadata sidecar per frame.
- Filename pattern: `shard-NNNNNN.tar` (zero-padded; padding width
  computed deterministically from expected shard count).
- Per-shard manifest: `_manifest.json` listing every member key and
  every clip key written into the shard.
- Output root: `MyDrive/fall_detection/artifacts/crops/`.
- Per-frame metadata sidecar carries the full contract needed by
  Pipeline A / B: `clip_id`, `dataset`, `label`, `source_path`,
  `track_id`, `frame_index`, `frame_offset`, `missing_frame`,
  `coverage`, `crop_config`, `margin_used`, `shard_filename`.

#### Issue 003 determinism contract

- Same input + same `CropConfig` → same crop pixels (byte-equal),
  same metadata bytes, same shard names.
- Provenance timestamps on every tar member are set to `mtime=0` so
  the tar bytes are reproducible across runs.
- WebDataset `cfg=` keys are not used here; the writer uses the
  stdlib `tarfile` module and Pillow / OpenCV for JPEG encoding.

## Issue 002 runtime fix — local-disk frame staging

### Root cause and symptom

Live Issue 002 runs showed identical ~1.8 fps on L4 and A100 GPUs.
The GPU was not the bottleneck — the runner was reading PNG frames
one at a time directly from the mounted Google Drive (a FUSE
filesystem). Drive small-file I/O is the documented bottleneck for
this project (PRD: "small-file I/O from Drive is prohibitively slow")
and FUSE reads can be cached arbitrarily, so the GPU sat idle
waiting on every frame.

### Fix

Each clip's frames are copied from Drive to `/content/fall_detection_local/
<clip_id>/` BEFORE tracking. The tracker reads from the local copy;
only the project's artefact paths on Drive (`artifacts/perception/`,
`logs/`) are written back. The raw dataset folder on Drive is never
modified. After tracking, the local copy is removed automatically by
the staging context manager so stale frames cannot contaminate the
next clip.

### Compute policy

**Issue 002 perception should run on a T4** with local-disk frames.
T4 + local disk is sufficient for YOLO26 + ByteTrack at
production-relevant FPS. **A100 is reserved for compute-bound work**,
especially VideoMAE fine-tuning in Issue 006 if it ends up needed.
Running Issue 002 on A100 does not help — we observed identical
~1.8 fps on both, because the bottleneck was I/O, not compute.

The notebook records `GPU_NAME`, `RUNTIME_TYPE` (T4 / A100 / L4 /
unknown), `torch.version.cuda`, and the local-staging root in the
perception run summary JSON.

### Extension to Issue 003 cropping

The Issue 003 cropping runner reads source PNG frames to build crop
tensors — same Drive FUSE small-file I/O bottleneck. The runner
now accepts an optional `local_root` parameter (default `None` for
backward compatibility; the 000 full-pipeline notebook passes
`COLAB_LOCAL_ROOT_DEFAULT`). When set, each clip's source frames
are copied to `<local_root>/crops_<clip_id>/` before any frame reads;
the local copy is removed via `try / finally` on clip exit so a
tracker exception can't leave stale frames behind.

The `RunSummary` payload now records `local_staging_root` so a
human reviewer can confirm the run used local staging. Set
`FALL_DETECTION_SKIP_LOCAL_STAGING=1` to opt out.

## URFD staging — read-only Kaggle source support

### New Kaggle / Colab behaviour

Newer Kaggle integrations expose datasets as a READ-ONLY mounted
input directory (e.g. ``/kaggle/input/<dataset>``) rather than a
writable cache at ``~/.cache/kagglehub/...``. The previous staging
code used ``shutil.move`` to relocate entries from the Kaggle source
into our staging root — that fails on a read-only mount (cross-device
rename errors, and the source-cleanup path can't delete the source).

### Fix

`data/stage_urfd.py` now uses a new `_copy_into(source, destination)`
helper that:

- Reads bytes from the source.
- Writes them into the destination via `shutil.copytree` (for dirs) or
  `shutil.copy2` (for files).
- **Never** deletes, renames, or otherwise mutates the source.

The Kaggle source is now treated as a read-only input — regardless of
whether the underlying mount is technically writable. The staging
helper creates the destination's parent directory if missing so a
single-file copy works without manual setup.

## Crop shard finalization — fixed

### Root cause (crop data correct, shards unterminated)

`run_cropping` discarded the `ShardWriter` returned by `_process_clip`
and never registered it for final close. The `open_writers` /
`_get_writer` plumbing was dead code (never called from the main
loop). The final close loop iterated over an empty dict, so:

- `_manifest.json` was never written (or was written empty).
- `summary.shards_written` was always 0.
- The tar files were left unterminated (writer entered via
  `__enter__`, never exited).

Crop *data* was being generated correctly — it was just never
sealed into a usable shard.

### Fix

`_process_clip` already returns the real writer on the success path
and a sentinel dummy (`/dev/null` writer, shard_index = -1) on the
empty-clip fallback. The new `run_cropping` loop:

1. Remembers the input shard index before calling `_process_clip`.
2. Calls `_process_clip` and gets `(outcome, next_shard_index, writer)`.
3. If `next_shard_index > pre_call_index` (real writer was created),
   registers the writer in `open_writers[pre_call_index]` so the final
   close loop can finalise it with its manifest.
4. Skips registration if the dummy was returned (shard_index
   unchanged, dummy points at `/dev/null`).

The dead `_get_writer` function has been removed — there is exactly
one writer per clip now (one writer per shard_index, created and
closed around the clip that produced it).

### Files changed

| Path | Change |
|---|---|
| `cropping/runner.py` | `run_cropping` now registers real writers from `_process_clip` in `open_writers`. Dead `_get_writer` removed. The final close loop + `summary.shards_written = len(open_writers)` correctly counts the actual shard count. Plus a small `with Image.open(...) as probe:` fix to silence a ResourceWarning. |
| `tests/test_run_cropping_end_to_end.py` | New: 1 end-to-end regression. Runs `run_cropping` on a real fixture, asserts windows emitted > 0, `shards_written` matches the real shard count, every shard tar contains a non-empty `_manifest.json`, and `read_shard()` round-trips the data. |

## Crop-stage path-resolution bug — fixed

### Root cause (real-run crash)

The first real Colab run of the cropping stage crashed because
`cropping/runner.py:_process_clip` resolved source frames against
the crop artefact root:

```python
source_folder = crops_root.parent / clip_record.source_path
```

This worked only when the dataset root and the crop artefact root
shared a parent — i.e. legacy single-root Drive layouts. In
LOCAL mode (Issue 002 → 003 → 005 bridge) the dataset root lives on
the Colab local disk (`/content/fall_local/`) and the artefact root
lives on Drive (`MyDrive/fall_detection/artifacts/crops`), so
`crops_root.parent` was `/content/drive/MyDrive/fall_detection/artifacts`
and the resolved path became the non-existent
`<artifact_root>/artifacts/datasets/urfd/<clip>`.

### Fix

`layout_root` is now a REQUIRED keyword parameter on `_process_clip`
and `run_cropping`. Source paths resolve against `layout_root /
clip_record.source_path` — the dataset/layout root, NOT the artefact
root. The previous `crops_root.parent` fallback has been removed:
it was always wrong in any layout where the crop root sat under
`artifacts/` (which is how `DriveLayout.ensure()` creates it).

`run_cropping`'s signature already required `layout_root`, so the
notebooks were already passing the right value — the bug was purely
inside `_process_clip`. The notebooks needed no change.

### Path contract (fixed)

| Resolution target | Root |

## First-class local-disk dataset mode (Issue 002 → Issue 003 → Issue 005 bridge)

### Root cause

Live runs with the manual hack
``FALL_DETECTION_DRIVE_ROOT=/content/fall_local SKIP_LOCAL_STAGING=1``
reduced runtime from ~30 minutes to ~2–4 minutes — confirming that
**Google Drive small-file I/O is untenable** for active processing.
The hack worked, but it was invisible to the notebooks themselves:
users had to know to set the env vars, and there was no logging that
confirmed the fast path actually fired.

### Decision

Local-disk dataset execution is now a **first-class pipeline mode**,
not an env-var hack. A new module `colab.data_mode` exposes:

- `DataMode.LOCAL` (default) — datasets land on the Colab local disk;
  artefacts still land on Drive so they survive a runtime reset.
- `DataMode.DRIVE` — legacy behaviour, both roots on Drive.
- `resolve_data_layout(mode=...)` returns a :class:`DataLayout` with
  separate ``dataset_root`` and ``artifact_root`` paths.
- `layout.root` alias points at ``dataset_root`` so existing cells
  that pre-date the data_mode split keep working unchanged.

The notebooks now do:

    DATA_MODE = DataMode(os.environ.get("FALL_DETECTION_DATA_MODE", "local").lower())
    layout = resolve_data_layout(mode=DATA_MODE)

…and log mode + both roots at the top so a human reviewer can confirm
the fast path is firing. **No manual env-var hacking required** —
users get the fast path by default. The opt-out
``FALL_DETECTION_SKIP_LOCAL_STAGING=1`` is documented as a per-clip
fine-grained escape hatch, not a prerequisite.

## Issue 002 review corrections

Issue 002 was code-complete but the review surfaced five blockers that
would have made the real Colab run fail or be unsafe. All five are
fixed; the code is now ready for a real Colab validation pass.

### 1. Nested URFD frame layout

URFD ships frames inside a wrapper folder::

    fall-01-cam0-rgb/
        fall-01-cam0-rgb/        <-- real frame folder
            frame_0001.png
            ...

`discover_frames` now detects this layout via
`_resolve_scan_folder` and descends into the single matching inner
folder automatically. `describe_layout` reports the layout as
`"flat"` or `"nested:<child>"` for diagnostic visibility. The numeric
sort is preserved across both layouts.

**Test coverage**: `test_urfd_nested_layout_descends_into_inner_matching_folder`,
`test_urfd_nested_layout_does_not_descend_when_inner_has_no_frames`,
`test_urfd_nested_layout_ignores_unrelated_subfolders`,
`test_describe_layout_reports_urfd_nested`.

### 2. Decode-failure observability

`_flatten_result` used to silently return an empty list on any decode
error — meaning a Ultralytics API change could produce a "successful"
run with zero detections on every frame, and the report would have no
idea anything was wrong.

Now:
- `_flatten_result` returns `(rows, decoded_ok)` — `decoded_ok=False`
  signals that the result object couldn't be interpreted.
- `run_tracker_on_folder` increments `result.decode_failures` on each
  undecodable frame.
- `decode_failures` is written into `run_meta.json` so a reviewer
  reading the artefact sees the signal immediately.
- A "decoded but no persons" frame still returns `(empty, True)` —
  no detections on a clean frame is not the same as a decode failure.

**Test coverage**: `test_filters_non_person_classes`,
`test_missing_boxes_is_a_decode_failure_not_a_silent_zero`,
`test_boxes_with_missing_attributes_is_a_decode_failure`,
`test_boxes_with_no_persons_is_decoded_ok_with_empty_rows`,
`test_decode_failures_appears_in_run_meta_artifact`.

### 3. `.pt` gitignore protection

`.gitignore` now excludes `*.pt`, `*.pth`, `*.onnx`, `*.bin`,
`*.safetensors`, `*.pkl`, `*.joblib`. Model weights (including
`yolo26m.pt`) downloaded by Ultralytics into the runtime cannot be
committed. The env lock captures versions but not the binary weights.

### 4. Dependency wiring confirmation

`kagglehub` is wired in BOTH `requirements.txt` (line 30) AND
`APPROVED_PIP_PACKAGES` in `colab/setup.py` (line 102). No reinstall
of torch / torchvision. Confirmed by `grep` in this session.

### 5. Fallback wiring corrected

The previous wiring forwarded `end2end=False` through
`cfg={"end2end": False}` — wrong, because `end2end` is a
**model/runtime constructor argument** to BoT-SORT, not a tracker
config key. The Ultralytics `cfg=` channel would have silently
ignored it, leaving the human reviewer with a misleading
"fallback applied" report.

After the fix:
- `fallback_track_low_thresh` is forwarded via
  `cfg={"track_low_thresh": ...}` — correct, both ByteTrack and
  BoT-SORT honour it.
- `fallback_tracker='botsort.yaml'` is forwarded via the top-level
  `tracker=` arg — correct.
- `fallback_end2end` is **NOT auto-wired**. The field stays in
  `TrackerConfig` so the *intent* is recorded in `run_meta.json`
  with the suffix `(manual intervention required)`. Applying it
  requires a manual code change (instantiating BoT-SORT directly).
- Defaults are unchanged: every fallback is `None` by default.
  **Do not run fallback levers unless the baseline real URFD result
  is weak** — Issue 002 rule, now enforced at the default config
  level.

**Test coverage**: `test_end2end_is_NOT_auto_wired_via_cfg`,
`test_end2end_alone_does_not_create_cfg`,
`test_end2end_is_marked_as_manual_intervention`,
`test_auto_wired_levers_appear_plain`.

## Issue 003 review corrections

Issue 003 was code-complete but two correctness bugs surfaced under
review that would have produced wrong crops in real Colab runs. Both
are fixed; 169 tests pass.

### 1. Edge-clipped crop geometry

**Bug:** when the expanded bounding box extended past an image edge,
`apply_crop_to_frame` sliced by `geometry.size` from `(x_min, y_min)` —
which over-extended the slice into pixels outside the visible image.
The bug only manifested on edge cases (a falling person near the
frame boundary — exactly the case we care about), so the existing
test suite missed it.

**Fix:** `CropGeometry` now carries `clipped_width` and `clipped_height`
fields explicitly. `square_with_padding` sets them to the actual
clamped dimensions of the input box. `apply_crop_to_frame` slices by
the clipped dimensions — never by `size` — so the slice is always
the exact rectangle inside the image. The canvas is still
`size × size` with symmetric pad bands on the un-clipped sides.

**Test coverage** (`tests/test_clip_builder.py:EdgeClippedCropTests`):

- `test_fully_in_bounds_box_unchanged` — regression guard for the
  happy path.
- `test_left_edge_clipped` — box runs off the left side.
- `test_right_edge_clipped` — box runs off the right side.
- `test_top_edge_clipped` — box runs off the top side.
- `test_bottom_edge_clipped` — box runs off the bottom side.
- `test_corner_clipped` — box runs off a corner (no pad needed).
- `test_two_edges_clipped_with_pad` — one clipped side wider than the
  other (real falling-person shape).
- `test_no_garbage_source_pixels_pulled_in` — sentinel buffer past
  the image edge; verifies the fix prevents the buggy slice from
  leaking memory garbage into the crop.

Each test asserts (a) the crop contains only pad + the expected
white-box pixels (no garbage), and (b) the clipped region is actually
narrower than the canvas when the box ran off an edge.

### 2. `track_windows.__all__` phantom exports

**Bug:** `track_windows.__all__` listed `MIN_BOX_AREA_PX`,
`MIN_BOX_SIDE_PX`, and `GAP_TOLERANCE` — none of which were defined
at module level. They were inlined as defaults in
`build_windows_for_track`, not exported.

**Fix:** removed the three phantom names from `__all__`. Cleaner than
adding the constants at module level: they are implementation
details of `build_windows_for_track`, not a public contract. Future
callers who want to read the defaults continue to look at the
function signature.

### 3. Optional `min_coverage_fraction` gate

**Status:** NOT applied. The existing `coverage` field on
`TrackWindow` already records the fraction of real vs missing frames;
the runner reads it for diagnostics. A separate `min_coverage_fraction`
gate would duplicate this surface and force every caller to reason
about two similar thresholds. Skipped per the optional / low-risk
constraint.

## Issue 005 Step 1 — VideoMAE-ready crop-shard loader

Pipeline A data-prep layer. Reads Issue 003 WebDataset-style
``.tar`` crop shards and assembles each clip / window into a
VideoMAE-ready ``(T, 3, H, W)`` float32 tensor, with provenance
and missing-frame mask preserved. No model code, no
perception code, no metrics — pure data prep.

### VideoMAE tensor contract

```
single clip: (T, C, H, W)
batch:       (B, T, C, H, W)
T ∈ {16, 32}     → ALLOWED_T              (default 16 → DEFAULT_T)
C = 3            → NUM_CHANNELS          (RGB)
H = W = 224      → IMAGE_SIZE             (HF videomae-base size.shortest_edge)
dtype            = float32
channel order    = RGB
```

The loader does NOT silently pad or truncate. A clip with
fewer or more than ``T`` frames surfaces as
`InvalidClipLengthError` ("too many") or `MissingFrameOffsetError`
("too few") — calling sites see the exact failure mode.

### Normalisation contract

The loader is pinned to HF `VideoMAEImageProcessor` defaults
for `"MCG-NJU/videomae-base"`:

```
image_mean = [0.485, 0.456, 0.406]    IMAGENET_MEAN
image_std  = [0.229, 0.224, 0.225]    IMAGENET_STD
size       = 224 (shortest_edge)      IMAGE_SIZE
```

The unit test
`test_local_constants_pinned_to_videomae_base` loads the
processor from the local HF cache and asserts the constants
match. A future processor-default drift trips the test
suite. The fallback path (`test_local_constants_are_documented_imagenet_values`)
guards against any environment where the HF cache is empty.

Pure-numpy normalisation pipeline
(`uint8 → [0, 1] → (x − mean) / std`) — no torch / torchvision
dependency, consistent with the Issue 001 review note that
torch is intentionally absent from the dev environment.

### Label / provenance contract

| Field | Source | Notes |
|---|---|---|
| `label` (string) | per-frame metadata | "fall" / "no_fall" — only legal values. `label_to_int` / `int_to_label` encode 0 / 1; anything else raises. |
| `clip_id`, `dataset`, `track_id`, `source_path` | per-frame metadata | enforced consistent across all frames in one clip — disagreement raises `InconsistentClipProvenanceError`. |
| `missing_frame` | per-frame metadata | preserved bit-for-bit on `LoadedClip.missing_mask`. |
| `coverage` | per-frame metadata | aggregated as the mean across frames on `LoadedClip.coverage`. |
| `missing_frame_count` | computed | int sum of `missing_mask`. |
| `frame_indices`, `frame_offsets` | per-frame metadata | preserved as numpy arrays / tuples. |
| `shard_filename` | (loader) | comma-joined list of source shards. |

### Required metadata fields

The loader rejects any clip whose `.meta.json` sidecar is
missing any of (raises `MissingMetadataFieldError`):

```
clip_id, dataset, label, track_id,
frame_index, frame_offset, source_path
```

### Frame-offset ordering

Frames are sorted by **numeric** `frame_offset`, not by tar
member name. The hard contract:

- Offsets are deduplicated before contiguity check.
- `frame_offsets` must equal `range(0, T)` exactly.
- Duplicate offsets raise `DuplicateFrameOffsetError`.
- Offsets missing from `0..T-1` raise `MissingFrameOffsetError`.

The "sort by numeric offset" rule is exercised against a tar
written with `offsets_in_storage_order = [9, 0, 5, 1, 4, 2, 7, 3, 6, 8, 10, 11, 12, 13, 14, 15]` —
the loader still produces a contiguous ``0..15`` clip sorted by
`frame_offset`.

### Missing-shard error message

`CropShardsMissingError(FileNotFoundError)` carries the
"Run Issue 003 first" hint at the front of the message:

```
Crop shard file(s) not found: /path/to/missing.tar.
Run Issue 003 (the cropping pipeline) first to produce shards,
or verify the path passed to load_clip_tensor_from_shards().
```

- No VideoMAE model load or fine-tune. (Issue 005 Step 2+.)
- No metric computation. (Issue 007 / 010 / 013 / 014 / 015.)
- No YOLO / ByteTrack / perception / crop-runner imports.
  This loader only reads shards Issue 003 already produced
  on disk.
- No torch / torchvision. Normalisation is pure numpy.

## Issue 005 Step 1 — loader grouping fix + full-pipeline notebook smoke check

### Bug description

The first cut of the Step 1 loader grouped frames by the bare
``metadata["clip_id"]`` field. Real Issue 003 shards, however,
store every window of one source clip under the **same bare
``clip_id``** — only the **per-window member stem**
(``<source>_t<track_id>_w<idx>``) distinguishes the windows.

Effect: a two-window source ``"X"`` produced 32 frames under bare
``"X"``; the loader assigned them all to one example; the
``T``-validations then surfaced a hard ``InvalidClipLengthError``
and the trainer was unusable on real shards. The test
``tests/test_load_crops.py::test_window_count_validates_per_window``
(renamed in the fix to ``MultiWindowShardTests`` /
``PrefixCollisionGuardTests``) would have caught this — but the
brief wired the fix before the regression test landed.

### Corrected ``clip_key`` contract

The contract is now spelled out on the module:

> ``clip_key`` is the **per-window member stem** written by
> Issue 003's runner — format ``<source_id>_t<track_id>_w<index>``
> (e.g. ``"X_t7_w000"``). Two windows from the same source clip —
> say ``"X_t7_w000"`` and ``"X_t7_w001"`` — share the same bare
> ``metadata["clip_id"]`` (= ``"X"``) but have **different** tar
> member stems. The loader groups frames by the per-window stem
> (prefix match using ``safe_member_name(clip_key) + "_"``); the
> bare source ``clip_id`` is preserved as ``LoadedClip.clip_id``
> for downstream manifest / role joins but is never the grouping
> key.

## Issue 005 Step 1 — window-key bug fix + T-adaptive smoke check

The Issue 005 smoke check immediately surfaced a window-key bug
in the crop runner: the writer name used
`f"{clip_record.clip_id}_t{track_id}_w{len(writer._clip_keys):03d}"`
inside the per-frame loop. ``len(writer._clip_keys)`` is a
*global* counter across all clips / tracks in a single shard —
it does not reset at clip boundaries. Every clip past the first
saw its windows numbered starting from the previous clip's
window count. The smoke check could not load the second clip's
windows because the keys it asked about did not exist on disk.

### Corrected contract

The crop runner now ``enumerate()``-s the emitted windows per
track and computes the per-window key **once per window**:

```python
for window_index, window in enumerate(build.emitted):
    clip_key = f"{clip_record.clip_id}_t{track_id}_w{window_index:03d}"
    ...
    for offset, (frame_idx, box) in enumerate(zip(window.frame_indices,
                                                  window.boxes)):
        ...
        writer.write_clip_member(clip_key, ...)
```

`clip_key` is stable across all frames in one window AND resets
to 0 at every clip / track boundary, so the per-window member
stems on Drive are now what the Pipeline A loader groups on
without manual fixup. ``window_index`` is also persisted on the
per-frame metadata (``metadata["window_index"]``) for downstream
provenance — a reviewer can recover a window's position without
re-parsing the manifest.

## Verifier — AST syntax check

### Bug description

`scripts/verify_notebook.py` previously relied on pyflakes to
catch malformed notebook code. Pyflakes' ``Reporter`` raises a
``StopIteration`` after the first ``SyntaxError`` it sees — but
when pyflakes is **not installed** (or installed at a version
that handles SyntaxErrors differently), the audit silently
passes broken cells. The audit's job is "fail loudly on real
wiring bugs", and a cell with an unterminated string literal
qualifies.

Two compounding real-world cases:

1. A future notebook edit introduces an unterminated string
   in the Pipeline A smoke check. Pyflakes catches it — until
   the audit is run on a host without pyflakes. The bug is
   silent in the "without pyflakes" path.
2. The audit's `errors` list groups findings by cell. Without
   a syntax check, a broken cell produces zero findings and the
   notebook prints "Status: OK" — making the issue invisible
   until a real Colab run.

### Syntax check contract

A new layer 1 (`ast.parse` per code cell) sits in front of the
existing local-import audit and pyflakes pass. It is the
load-bearing gate:

- **Independent of pyflakes.** A notebook with an unterminated
  string literal always fails when pyflakes is missing.
- **Per-cell reporting.** Each broken cell produces one
  :class:`scripts.verify_notebook.CellFinding` whose
  ``errors`` tuple contains a single ``SyntaxError`` message
  with line and column number, the parser's own ``msg`` text,
  and the offending source line. The output format matches the
  existing two layers so the CLI summary is consistent.
- **CLI exit code is non-zero** when any cell has a SyntaxError.
  ``Status: OK — notebook is ready for Colab`` is forbidden when
  a syntax check fires.
- **Existing behaviour preserved.** The local-import audit and
  the cross-cell ``undefined name`` warning surface — neither
  is removed or changed in semantics.

Implementation:

```python
def _check_cell_syntax(source: str) -> list[str]:
    """Return one error string per Python SyntaxError in ``source``.

    Uses ``ast.parse`` (stdlib) so the check works even when
    pyflakes is not installed.
    """
    import ast
    try:
        ast.parse(source, filename="<notebook-cell>")
        return []
    except SyntaxError as exc:
        line = exc.lineno or 0
        col = exc.offset or 0
        bad_line = (exc.text or "").rstrip("\n")
        msg = (
            f"Python SyntaxError at line {line}, col {col}: "
            f"{exc.msg}"
        )
        if bad_line:
            msg += f"  (line: {bad_line.rstrip()!r})"
        return [msg]
```

## URFD university CSV label parser — Step 2

The Issue 005+ pipeline consumes the two authoritative
university URFD cam0 label files:

- `urfall-cam0-falls.csv` — per-frame labels for fall-01..fall-30.
- `urfall-cam0-adls.csv` — per-frame labels for adl-01..adl-40.

The parser produces a validated, ready-to-query lookup. The
loader can call `is_contiguous(sequence)` before training a
window on a sequence to confirm every frame is labelled —
catching half-staged trees before the label silently mismatches
the frame index.

### CSV format (verified)

```
<sequence>,<frame_number>,<label>,<col4>,<col5>,...,<col11>
```

- **No header.**
- **11 columns**; the parser uses only the first 3.
- Column 1: ``sequence`` — e.g. ``"fall-01"`` / ``"adl-01"``.
- Column 2: ``frame_number`` — 1-based integer (the clip's
  intra-clip frame index, NOT a global video timestamp).
- Column 3: ``label`` — one of ``{-1, 0, 1}``.
- Columns 4..11: ignored. The real university CSVs carry
  per-frame bounding-box / sub-class metadata that the project
  does not consume.

### Label meanings

| Value | Meaning | Use case |
|---|---|---|
| ``-1`` | Upright | No fall in progress. Default state of every adl clip and the pre-fall / post-fall regions of every fall clip. |
| ``0`` | Falling / transition | The fall is happening — the subject is mid-air / mid-fall. |
| ``1`` | Lying on the ground | The fall has completed; the subject is on the floor. |

`VALID_LABELS = frozenset({-1, 0, 1})`. `LABEL_MEANINGS` is the
public run-summary string for each value.

### Frame-number alignment assumption

The brief says: frame numbers are 1-based integers. The
parser enforces this — non-positive frame numbers raise
`MalformedURFDLabelRow`. The contiguity check
`is_contiguous(sequence)` is the load-bearing alignment
guarantee: it returns `True` iff the parsed frames form a
contiguous 1..N run, no gaps. A clip whose CSV has frame 5
missing surfaces as `is_contiguous("fall-01") == False`
**before** the loader builds a training example on it.

### Failure modes

Every malformed row raises `MalformedURFDLabelRow` (a
`ValueError` subclass) with a message that includes the source
path and the line number:

- Fewer than 3 comma-separated columns.
- Empty sequence.
- Non-integer frame number.
- Non-positive frame number (`<= 0`).
- Non-integer label.
- Label outside `{-1, 0, 1}`.
- Duplicate frame number within the same sequence.

Blank lines and lines containing only whitespace are skipped
(not malformed). The first 3 columns of every other non-empty
line are validated.

### Files created

| Path | Purpose |
|---|---|
| `data/urfd_labels.py` | `FrameLabel`, `CSVLabels`, `parse_urfd_csv_label_text`, `parse_urfd_csv_label_file`, `MalformedURFDLabelRow`, `VALID_LABELS`, `LABEL_MEANINGS`. |
| `tests/test_urfd_labels.py` | 26 unit tests: small-fixture happy path, mixed fall+adl sequence, real-archive 160-row alignment anchor, every malformed-row type, contiguity predicate, container shape. |

### Public API contract

```python
parsed = parse_urfd_csv_label_file(
    Path("urfall-cam0-falls.csv"),
)
# or
parsed = parse_urfd_csv_label_text(text, source_label="...")

# Structured lookup
parsed.lookup(sequence, frame_number)         # int label or None
parsed.frame_count(sequence)                  # int
parsed.frame_range(sequence)                  # (min, max) inclusive
parsed.is_contiguous(sequence)                # True iff 1..N with no gaps
parsed.sequences()                            # tuple[str, ...] sorted
```

The :class:`FrameLabel` and :class:`CSVLabels` dataclasses
validate at construction time — a hand-built invalid record
is rejected the same way a malformed CSV row would be.

## URFD window labeling — Step 3

The Issue 005+ pipeline now turns authoritative per-frame
URFD labels into clean window labels at training time. The
labels are NOT baked into crop shards — a future rule change
swaps the labelling without re-cropping.

### Public signature

```python
from data.urfd_labels import (
    CSVLabels,                     # from Step 2
    DefaultWindowLabelingRule,     # default rule
    WindowLabelingRule,            # base class for swap-ins
    WindowLabelingError,
    WINDOW_LABEL_FALL, WINDOW_LABEL_NO_FALL,
    clip_id_to_sequence,
    label_window,
)

label, is_confuser = label_window(
    clip_id="urfd-debug-fall-01-cam0-rgb",
    frame_indices=[0, 1, ..., 15],   # REAL crop indices are 0-based
    csv_labels=parsed,
    frame_index_offset=1,            # REQUIRED for real crops: 0-based
                                     # crop frame_index → 1-based CSV.
                                     # See "Frame-index alignment" below.
    # labeling_rule=DefaultWindowLabelingRule() — Issue 006 can
    # pass a different subclass to swap the rule.
)
```

`label` is one of :data:`WINDOW_LABEL_FALL` /
:data:`WINDOW_LABEL_NO_FALL` (strings, matching the rest of
the pipeline). `is_confuser` is a bool flag the loader can
filter on.

### Default rule

Fall clips:
    - if any frame in the window has label ``0`` (falling) or
      ``1`` (lying), the window is **fall**.
    - if every frame in the window is ``-1`` (upright), the
      window is **no_fall** — the pre-fall region of a fall
      clip is a clean negative, NOT a noisy positive.

ADL clips:
    - always **no_fall** (the source clip type is non-fall).
    - if any frame in the window has label ``1`` (lying), the
      window is flagged ``is_confuser=True`` so a downstream
      training run can choose to mix it into the no-fall pool,
      weight it down, or drop it. The default keeps the flag on
      so the evaluation can study confuser-aware behaviour
      without further label surgery.

### Frame-index alignment decision

**RESOLVED (commit 8188142, re-verified b9578a2): crop
frame_index is 0-based; URFD CSV is 1-based; real callers pass
`frame_index_offset=1`.** Verified against the actual code:

- `perception/tracker.py:403` uses `enumerate(stream)` with no
  `start=`, so detection `frame_index` is **0-based**.
- `cropping/runner.py:130,181` reads `raw["frame_index"]` and
  writes it to `meta.json` unchanged — **no `+1` anywhere** in
  the perception → crop chain. So the crop sidecar's
  `"frame_index"` is 0-based.
- URFD CSV `frame_number` is 1-based (RGB filenames also 1-based,
  `fall-01-cam0-rgb-001.png`).

Therefore real crop windows must be joined to CSV labels with
`frame_index_offset=1`. The live notebook smoke
(`000_full_pipeline.ipynb`) already does this.

RESOLVED (commit 09b4a56): the false docstrings are corrected and
a correct-by-construction real-crop entry point now exists.
`data/urfd_labels.py` exports `CROP_TO_CSV_FRAME_OFFSET = 1` and
`label_window_from_crop_meta(clip_id, crop_frame_indices, csv,
*, labeling_rule=...)`, which applies the `+1` internally. **Every
real-crop caller (Issue 006 training/eval, the notebook smoke)
MUST use `label_window_from_crop_meta` — NOT the primitive
`label_window`**, whose default `frame_index_offset=0` is a
no-shift knob kept only for synthetic fixtures. A regression that
drops the `+1` is caught by
`test_wrapper_plus_one_is_load_bearing_at_fall_onset_boundary`
(a 51..82 window that is `fall` only because the offset is
applied). Suite: 678 pass.

### Guard behavior

Every malformed input raises `WindowLabelingError` (a
`ValueError` subclass) with a message that identifies the
specific problem:

- `frame_indices` is empty.
- `clip_id` does not match the expected
  `urfd-debug-<sequence>-cam0-rgb` shape.
- The mapped sequence's CSV labels are not contiguous
  (`CSVLabels.is_contiguous` returns `False`). A
  half-staged tree must not produce clean training examples.
- Any frame index in the window, after `frame_index_offset` is
  applied, is missing from the CSV labels.
- A non-integer frame index in the caller's list.
- A non-positive adjusted frame index.

### Pluggable rule

`WindowLabelingRule` is an abstract base with one method:

```python
class WindowLabelingRule:
    def apply(self, clip_type, per_frame_labels) -> tuple[str, bool]:
        raise NotImplementedError
```

The default rule is `DefaultWindowLabelingRule`. Issue 006 can
construct a different subclass (e.g. a stricter rule that only
credits `label == 1` as fall) and pass it to `label_window`
without re-cropping. The test suite proves this with
`StrictFallbackRule` — the same window under a stricter rule
returns `no_fall` where the default returns `fall`.

## URFD window labeling — sparse-label fix

### Finding

Real university ADL CSVs are sparse: some RGB frames have
no corresponding CSV label row. The Step 3 first cut raised
`WindowLabelingError` on a missing frame and on a
non-contiguous sequence — both legitimate states of a real
sparse ADL CSV. The first run on a real ADL cluster crashed
inside the loader: the first window of an ADL clip asked for
labels on frames 1..16, the CSV only labelled frames 6..150,
the function raised, the pipeline stopped.

### Revised missing-label contract

:func:`label_window` no longer treats sparse labels as a
bug:

- The `is_contiguous(sequence)` gate is removed. A
  half-staged tree is no longer a hard stop; the helper
  remains on `CSVLabels` for callers who want to audit the
  CSV independently.
- A missing frame index in the window's `frame_indices`
  is **silently skipped** rather than raised. Real ADL
  CSVs skip frames; the function collects only the labels
  that exist and passes them to the labelling rule.
- A non-contiguous CSV no longer blocks labelling. The
  `is_contiguous` helper is informational; it is not a
  guard.

### New sentinel: `WINDOW_LABEL_UNLABELED = "unlabeled"`

The default rule's zero-availability paths return explicit
outcomes rather than `("no_fall", False)` (which would
silently treat "no labels" as "all negative"):

- **Fall clip, zero available labels** → the sentinel
  `WINDOW_LABEL_UNLABELED`. Issue 006 can drop the
  example deterministically; the sentinel is a third
  state distinct from both `fall` and `no_fall`.
- **ADL clip, zero available labels** → `(WINDOW_LABEL_NO_FALL,
  False)`. A zero-label ADL window is still "no_fall by
  source-clip type"; the confuser flag requires a positive
  label-1 signal, which a sparse label set cannot produce.

### Guard behaviour (unchanged for genuine bugs)

`WindowLabelingError` is still raised, but ONLY for genuine
bugs:

- `frame_indices` is empty.
- `clip_id` does not match `urfd-debug-<sequence>-cam0-rgb`.
- A non-integer frame index in the caller's list.
- A non-positive adjusted frame index.

A missing frame or a non-contiguous CSV is **not** a failure
mode. The function handles both silently and continues.

## Issue 006 Prompt #1 — crop-availability guard fix

Pre-acceptance review of `colab/006_pipeline_a_videomae.ipynb` caught a blocking bug in the §2 crop-availability guard: it derived the **41 fall / 199 no-fall** window-count check from `meta["label"]`, but `cropping/runner.py` writes that field from `clip_record.label` — i.e. it is **source-clip provenance**, not a window-level fall/no_fall decision. The exact 41/199 numbers in context.txt come from that clip-level breakdown (30 fall clips × ~1-2 windows ≈ 41; 40 ADL clips × ~5 windows ≈ 199), not from a CSV-derived window-level pass.

The fix uses the **stronger correct check** the brief offered: compute the window-level label for every window via `label_window_from_crop_meta(...)` (the supported real-crop entry point) and verify a small set of **consistency invariants** rather than an exact-count match. The exact window-level fall count depends on which windows span the falling/lying region vs. the pre-fall region of each fall clip — a number we do not know a priori — so an exact-count check would either be over-specified or would require a separate audit step.

### Decisions

1. **`meta["label"]` is source-clip provenance only.** The guard records the per-source-label window count for diagnostic transparency but does NOT compare it against a fixed expected window-level-fall number.
2. **Window-level labels come exclusively from `label_window_from_crop_meta(...)`** against the persisted URFD CSVs. Sub-second for 240 windows × ~32 frames each.
3. **Invariants checked (cheap, sufficient, and correct):**
   - every window resolves to one of `fall` / `no_fall` / `unlabeled`
   - `fall_count + no_fall_count + unlabeled_count == total_windows`
   - **no ADL-source (no_fall source) window resolves to `fall`** — that would be a CSV / offset wiring bug
   - shard count, total window count, unique-clip count, and clip-id shape (all in expected bands; ~70 / ~240 / ~70 / `urfd-debug-<seq>-cam0-rgb`)
4. **Tiny-slice overfit criterion loosened** from the brittle absolute `final_loss > 0.30` to a two-part check:
   - every loss is **finite** (no NaN / Inf), AND
   - `final_loss ≤ 0.5 × initial_loss` (substantial relative decrease)
   This catches both "model didn't learn at all" and "loss exploded" without depending on random init weights.
5. **Frozen-tier guard made real** by replacing the prefix-string check with `data.manifests.load_manifest` + `select_clips_for_context(manifest, ExecutionContext.EVALUATION)` + `enforce_no_frozen_in_iterable(manifest.clips, ctx)`. The notebook now loads the URFD manifest from `layout.dataset_root/datasets/urfd/manifest.yaml`, computes the EVALUATION-allowed clip set, and asserts every window's `clip_id` lives inside it. Today this is trivially satisfied (URFD is all `DEBUG`); the guard catches a future regression where someone accidentally points the notebook at a vault dataset (omnifall / caucafall / mcfd / fallvision).

## Issue 006 Prompt #2 — frozen-backbone VideoMAE fine-tune

This prompt stands up **rung-1** of Pipeline A: cached-feature extraction from a frozen Kinetics-finetuned VideoMAE backbone, followed by linear-head training on real validation. The backbone is **not** unfrozen at this prompt — that is the explicit scope guardrail.

### Decisions

1. **Primary model**: `MCG-NJU/videomae-base-finetuned-kinetics` (Kinetics pre-training signal that the frozen backbone carries). Plain `MCG-NJU/videomae-base` is kept only as an **ablation note** in §3 — not the main path.
2. **Trustworthy model load**: the Kinetics-finetuned state dict is downloaded separately and every non-classifier key is asserted to match the loaded model exactly (shape + presence). Classifier head is allowed to differ because of `num_labels=2 + ignore_mismatched_sizes=True`; backbone mismatches fail loud. The model is pinned to the HF Hub id and the load path is recorded in the run log (`metrics/pipeline_a/006_pipeline_a_run_log.json`).
3. **Stratified source-clip split**: clip_ids are partitioned by source family (`fall-*` vs `adl-*`), then each family is deterministically shuffled and split 70/15/15 at seed=42. The validation set is **asserted** to contain at least one fall-source window. True window-level fall / no_fall / unlabeled counts per split are reported using `label_window_from_crop_meta(...)` — NOT the source-clip label.
4. **Validation baselines before any training**:
   - **Majority-class all-no_fall** (every prediction = 0.0). AUPRC = positive rate (~0.17 on the current val set).
   - **Frozen Kinetics + untrained binary head** (freshly-reinitialised 2-class head). AUPRC near 0.5.
   Both are run through the SAME `compute_classification_metrics` harness the trained model uses — apples-to-apples comparison.
5. **Rung-1 training as frozen-backbone feature extraction + linear head**:
   - Backbone frozen (`requires_grad=False` on every non-classifier parameter).
   - Features extracted by `model.video_mae(pixel_values=...)` in `eval()` + `no_grad()`. Output is `last_hidden_state[:, 0, :]` (the `[CLS]` token — the natural input to the original classifier head).
   - AMP capability-detected: `bf16` on L4/A100 where supported, `fp16` on T4.
   - Features cached in memory (~740 KB total — trivial).
   - Peak GPU memory captured via `torch.cuda.max_memory_allocated()`.
   - Fresh `nn.Linear(HIDDEN_SIZE, 2)` trained on cached features with **weighted cross-entropy** (inverse-frequency class weights on train window-level labels). AdamW, `lr=1e-3`, `weight_decay=1e-4`, manual loop, 30 epochs, batch size 16.
   - **Do NOT reuse the §7 tiny-slice overfit model** — the §8 cell reloads the model fresh.
6. **Checkpointing + resume**: every epoch whose `val_auprc > best_auprc` writes `<artifact_root>/checkpoints/pipeline_a/linear_head.pt` with `{epoch, head_state, optimizer_state, best_val_auprc, split_seed, class_weights, hidden_size, model_id, transformers_version, torch_version}`. On re-run, `_load_latest` reads the checkpoint and resumes from `start_epoch + 1`. The checkpoint lives under `artifacts/checkpoints/` which `.gitignore` already excludes.
7. **Epoch selection: best val AUPRC only — never accuracy.** The selection key is `best_val_auprc` in the checkpoint.
8. **Validation evaluation through the harness**:
   - **Probabilities come from `softmax(logits)[:, 1]`** — NEVER raw logits, NEVER `argmax`.
   - **Window-level** predictions + labels: `ClipPrediction(score=fall_score, ...)` + `ClipLabel(label=FALL/NO_FALL, ...)` per window.

### Fix 3 — Feature pooling matches the actual classifier path

VideoMAE has **no CLS token**. The original `.classifier` takes `last_hidden_state[:, 0, :]` because index 0 happens to be the first patch token (a temporal tubelet), not a learned class token. A fresh linear head trained on `last_hidden_state[:, 0, :]` is wired to the *wrong* representation.

The pooling is now:

```python
out = model.videomae(pixel_values=pixel_values)
hidden = out.last_hidden_state
pooled = hidden.mean(dim=1)                       # mean-pool over all patch tokens
if getattr(model, "fc_norm", None) is not None:   # BatchNorm before the head, if present
    pooled = model.fc_norm(pooled)
feat = pooled.float()
```

Applied to **both** the feature-cache path (§9) and the FPS / timing path (§11) so the two paths cannot drift. The comment that claimed a CLS token was removed.

Grep audit:

```

```

### Bonus (low-risk, optional) — checkpointing + AMP

- **Save both `linear_head_best.pt` and `linear_head_latest.pt`** every epoch (not just best). The `latest` is for resume on crash; the `best` drives the metric for Issue 007. A new `_save_ckpt_to(path, ...)` helper makes the two paths explicit; `_save_ckpt(...)` is kept as a backward-compat wrapper that writes to the best path.
- **AMP dtype selection**: prefer `fp16` on T4, keep `bf16` on L4/A100 where supported, fall back to `fp16` on older hardware. The previous logic took `bf16` whenever `torch.cuda.is_bf16_supported()` returned True, which on some T4 driver combinations silently downgrades to a slow bf16 path. The new selection uses the `gpu_name` recorded in §1a to break the tie.
