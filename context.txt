## Issue 002 decisions and rules

### 1. URFD source and staging

- **Source.** `tanmaydacha/urfd-dataset` on Kaggle. Whitelisted in
  `data/stage_urfd.py:ALLOWED_KAGGLE_SLUG`. Any other slug raises before
  any network call.
- **Destination.** `MyDrive/fall_detection/datasets/urfd/`.
- **Idempotency.** A marker file `.staged_from_kaggle.txt` is written
  on success; re-runs short-circuit. `force=True` overrides for the
  rare case where the staged tree is corrupted.
- **Provenance.** The marker records the slug and the credential source
  ("colab_secrets") — never the credential values themselves.

### 2. Kaggle credential handling rule

- **Read from Colab Secrets at runtime.** `KAGGLE_USERNAME` and
  `KAGGLE_KEY` from `google.colab.userdata`.
- **Never** print, log, return, commit, or write the values anywhere.
  The staging script's only credential-aware code path sets two env
  vars and returns; the values never leave the function.
- **Local-dev fallback.** Outside Colab, the script looks for
  `~/.kaggle/kaggle.json` on disk. We do NOT auto-create it.
- **No fallback to env vars or files outside the two paths above.**
  This keeps the credential surface tiny and auditable.

### 3. Folder-of-frames input contract

The tracker accepts an **ordered** iterable of frame paths:

- The caller (the notebook / orchestrator) is responsible for ordering.
- `FrameFolderReader` enforces numeric sort by trailing integer in the
  filename stem (`frame_0001.png` → 1, `fall-01-cam0_frame_0042.png`
  → 42). Non-numeric frames sort last by filename.
- Out-of-order frames produce silently invalid tracks. The bug is
  invisible until you visualise the annotated output. **Never bypass
  the loader.**

### 4. Numeric frame sorting requirement

- Implemented in `perception/frames.py:discover_frames`.
- Sort key: `(has_number_flag, trailing_integer, filename)`.
- Frames with a trailing integer sort by that integer ascending.
- Frames without one sort to the END (so they don't poison the start
  of the timeline).
- Ties on the integer (rare) are broken by full filename (lexical) for
  determinism.

### 5. YOLO26 + ByteTrack configuration

| Knob | Value | Source of truth |
|---|---|---|
| `model_name` | `yolo26m` | `perception/tracker.py:REQUIRED_MODEL` |
| `tracker_config` | `bytetrack.yaml` | `perception/tracker.py:REQUIRED_TRACKER` |
| `persist` | `True` | (required for ByteTrack ID survival across frames) |
| `classes` | `[0]` (person) | `perception/tracker.py:PERSON_CLASS_ID` |
| `conf` | `0.25` | `TrackerConfig` default |

**Hard rule:** any model name other than `yolo26m` raises
`UnsupportedModelError`. No silent fallback to yolo11 / yolov8 / etc.
Pretrained weights only — no training in this module.

### 6. Metrics limitation due to missing ground truth

URFD ships with raw frames only — no detection or tracking labels.
The report therefore does NOT compute mAP, IDF1, MOTA, or HOTA. They
are recorded in `metric_availability` as:

```python
"map_50":     "n/a (no detection ground truth)"
"map_50_95":  "n/a (no detection ground truth)"
"idf1":       "n/a (no tracking ground truth)"
"mota":       "n/a (no tracking ground truth)"
"hota":       "n/a (no tracking ground truth)"
```

Debug-tier validation is **annotated visual review** + the
track-continuity / fragmentation report. The PRD already calls this
out as the "cheapest killer risk to retire first" — track-through-fall
must survive the fall motion itself.

### 7. Fallback levers (apply only if the baseline fails)

In escalation order, exposed on `TrackerConfig`:

1. `fallback_track_low_thresh` — lower ByteTrack's association threshold
   (e.g. `0.1`) so weak detections still get associated. **Auto-wired**
   via Ultralytics ``cfg={"track_low_thresh": ...}`` — ByteTrack and
   BoT-SORT both honour this key.
2. `fallback_tracker='botsort.yaml'` — switch tracker config to BoT-SORT
   (different association algorithm). **Auto-wired** via
   ``tracker=`` arg.
3. `fallback_end2end=False` — disable the BoT-SORT end-to-end pass.
   **NOT auto-wired.** ``end2end`` is a model/runtime constructor arg of
   the BoT-SORT class, not a tracker config key — it cannot be expressed
   through ``model.track()``'s ``cfg=``. The field is kept on
   ``TrackerConfig`` so the *intent* is recorded in ``run_meta.json``
   (with the ``manual intervention required`` annotation); applying it
   requires a manual code change.

Whichever lever was used is recorded in ``run_meta.json`` as
``fallback_used``. Auto-wired levers appear bare; manual-only levers
appear with the ``(manual intervention required)`` suffix so a human
reviewer can tell at a glance what was actually applied.

Default ``TrackerConfig()`` has every fallback set to ``None`` — the
baseline is what runs by default. **Do not run fallback levers unless
the baseline real URFD result proves weak** (Issue 002 rule).

### 8. URFD manifest shape

- Schema version: `1.1`.
- All clips are `role: debug` (URFD is a debug-tier dataset).
- Labels:
  - `fall-NN-camM` → `fall`
  - `adl-NN-camM` → `no_fall`
- `source_path` is **Drive-relative**: `datasets/urfd/<folder_name>`.
- `notes` carries `sequence=NN; camera=camM; frame-folder (PNGs)`
  for downstream traceability.

## Open issues / not yet addressed

- **Track-through-fall has not been visually confirmed yet.** The
  perception front-end runs end-to-end and writes artefacts, but the
  actual question — *does the falling person keep the same track_id
  through the fall motion on real URFD footage?* — is a human review
  question answered by opening the annotated frames on Drive. If the
  answer is "no, ID switches happen at the moment of collapse on most
  fall clips," we escalate via fallback levers before moving to Issue
  003. **Decision deferred until a real Colab run is reviewed.**
- **Issue 002 is NOT closed.** The code is complete and verified
  (116 tests pass) but the closure condition is a real Colab run on at
  least one `fall-*` URFD clip with human-reviewed annotated output
  showing stable ID through the fall. Until that visual review passes,
  Issue 002 stays in "code-complete, awaiting closure". See
  *Issue 002 review corrections* below.
- **mAP / IDF1 / MOTA / HOTA are not computable on URFD.** Recorded
  as "n/a" rather than fabricated. When a dataset with tracking
  ground truth arrives (none currently on the PRD list), Issue 004
  eval harness can compute them.
- **Real URFD has not been staged in this dev environment.** Only the
  placeholder manifest and the staging script are in the repo. A real
  Colab session needs to run `colab/002_perception_urfd.ipynb` once
  with Kaggle Secrets configured.
- **`MAX_FRAMES_PER_CLIP=120`** in the notebook is a debug-tier cap.
  Raise it (or remove it) when the wiring is trusted and we want
  full-length runs.
- **Annotator hasn't run on the full URFD sequence set.** Real URFD
  has 30 fall sequences × 2 cameras + 40 ADL sequences × 2 cameras.
  The placeholder + test fixtures cover 4 sequences; the staged real
  dataset will expand to the full set.
- **Cross-listed URFD clips in train/validate.** URFD currently lives
  only in `debug` per the Issue 001 manifest schema (URFD was listed
  in the debug-tier decisions). If Issue 006/009 needs URFD for
  training, that's a separate decision — Issue 001 review already
  recorded "train/validate disjointness depends on real subject_id
  population quality", and URFD lacks subject IDs entirely.

## Issue 002 review corrections

Issue 002 was code-complete but the review surfaced five blockers that
would have made the real Colab run fail or be unsafe. All five are
fixed; the code is now ready for a real Colab validation pass.

### 1. Nested URFD frame layout

URFD ships frames inside a wrapper folder::

    fall-01-cam0-rgb/
        fall-01-cam0-rgb/        <-- real frame folder
            frame_0001.png
            ...

`discover_frames` now detects this layout via
`_resolve_scan_folder` and descends into the single matching inner
folder automatically. `describe_layout` reports the layout as
`"flat"` or `"nested:<child>"` for diagnostic visibility. The numeric
sort is preserved across both layouts.

**Test coverage**: `test_urfd_nested_layout_descends_into_inner_matching_folder`,
`test_urfd_nested_layout_does_not_descend_when_inner_has_no_frames`,
`test_urfd_nested_layout_ignores_unrelated_subfolders`,
`test_describe_layout_reports_urfd_nested`.

### 2. Decode-failure observability

`_flatten_result` used to silently return an empty list on any decode
error — meaning a Ultralytics API change could produce a "successful"
run with zero detections on every frame, and the report would have no
idea anything was wrong.

Now:
- `_flatten_result` returns `(rows, decoded_ok)` — `decoded_ok=False`
  signals that the result object couldn't be interpreted.
- `run_tracker_on_folder` increments `result.decode_failures` on each
  undecodable frame.
- `decode_failures` is written into `run_meta.json` so a reviewer
  reading the artefact sees the signal immediately.
- A "decoded but no persons" frame still returns `(empty, True)` —
  no detections on a clean frame is not the same as a decode failure.

**Test coverage**: `test_filters_non_person_classes`,
`test_missing_boxes_is_a_decode_failure_not_a_silent_zero`,
`test_boxes_with_missing_attributes_is_a_decode_failure`,
`test_boxes_with_no_persons_is_decoded_ok_with_empty_rows`,
`test_decode_failures_appears_in_run_meta_artifact`.

### 3. `.pt` gitignore protection

`.gitignore` now excludes `*.pt`, `*.pth`, `*.onnx`, `*.bin`,
`*.safetensors`, `*.pkl`, `*.joblib`. Model weights (including
`yolo26m.pt`) downloaded by Ultralytics into the runtime cannot be
committed. The env lock captures versions but not the binary weights.

### 4. Dependency wiring confirmation

`kagglehub` is wired in BOTH `requirements.txt` (line 30) AND
`APPROVED_PIP_PACKAGES` in `colab/setup.py` (line 102). No reinstall
of torch / torchvision. Confirmed by `grep` in this session.

### 5. Fallback wiring corrected

The previous wiring forwarded `end2end=False` through
`cfg={"end2end": False}` — wrong, because `end2end` is a
**model/runtime constructor argument** to BoT-SORT, not a tracker
config key. The Ultralytics `cfg=` channel would have silently
ignored it, leaving the human reviewer with a misleading
"fallback applied" report.

After the fix:
- `fallback_track_low_thresh` is forwarded via
  `cfg={"track_low_thresh": ...}` — correct, both ByteTrack and
  BoT-SORT honour it.
- `fallback_tracker='botsort.yaml'` is forwarded via the top-level
  `tracker=` arg — correct.
- `fallback_end2end` is **NOT auto-wired**. The field stays in
  `TrackerConfig` so the *intent* is recorded in `run_meta.json`
  with the suffix `(manual intervention required)`. Applying it
  requires a manual code change (instantiating BoT-SORT directly).
- Defaults are unchanged: every fallback is `None` by default.
  **Do not run fallback levers unless the baseline real URFD result
  is weak** — Issue 002 rule, now enforced at the default config
  level.

**Test coverage**: `test_end2end_is_NOT_auto_wired_via_cfg`,
`test_end2end_alone_does_not_create_cfg`,
`test_end2end_is_marked_as_manual_intervention`,
`test_auto_wired_levers_appear_plain`.

### 6. Issue 002 closure condition

The remaining condition for closing Issue 002 is NOT code — it's a
real Colab run:

> Run `colab/002_perception_urfd.ipynb` end-to-end with real Kaggle
> credentials on at least one `fall-*` URFD clip. Open the annotated
> frames at
> `MyDrive/fall_detection/artifacts/perception/<clip_id>/annotated/`
> on Drive and visually verify that the falling person keeps the same
> track_id through the fall motion.

If that visual review passes, Issue 002 is closed and Issue 003 can
start. If it fails (ID switches at the moment of collapse on most
fall clips), apply fallback levers in order before retrying:
`fallback_track_low_thresh=0.1` first, then `fallback_tracker='botsort.yaml'`,
then `fallback_end2end=False` (manual).

## Assumptions

- **ultralytics 8.4+** is installed in the runtime; `yolo26m` is
  available. Verified against the dev environment in this session.
- **PyYAML** is available for the manifest builder (in the approved
  stack).
- **Pillow** is available for the renderer (in the approved stack).
- **kagglehub** reads `KAGGLE_USERNAME` / `KAGGLE_KEY` from env vars.
  This is the documented contract.
- **URFD PNG frames are named with a trailing integer** (e.g.
  `frame_0042.png`). If a future URFD release changes naming, the
  loader's regex may need to be widened — flagged for review.
- **Colab Secrets are persistent across sessions for a given user.**
  They are not stored on Drive.
