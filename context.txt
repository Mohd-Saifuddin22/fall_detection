## Issue 001 review corrections (this revision)

The following corrections were folded in after the first pass. Each one
addresses a specific acceptance-criterion gap called out in the review.

### 1. Leakage enforcement is now complete

The previous train/validate disjointness check only triggered on
duplicate `clip_id`. Two additional checks now run:

- **`_check_source_path_disjoint`** — train and validate must not share
  the same `source_path`. Guards against the realistic failure mode where
  two annotators assign different `clip_id`s to overlapping windows of
  the same raw video.
- **`_check_subject_id_disjoint`** — train and validate must not share
  the same `subject_id` when one is set. Guards against subject-level
  leakage where the model memorises person-specific cues instead of
  learning the fall pattern.

Three new test classes exercise both rules: `SourcePathDisjointTests`,
`SubjectIdDisjointTests`, plus the original `DisjointTrainValidateTests`
for the duplicate-clip_id case.

**Caveat (open issue).** Subject-level leakage enforcement depends on
real `subject_id` population quality. When subject_id is `None` on both
sides of a comparison, the rule has nothing to compare and silently
passes. The check is correct, but it cannot save a manifest whose
authors forgot to populate subject_id. We will tighten this if / when
dataset authors are willing to commit subject IDs at scale. (See the
*Open issues* section below.)

### 2. Optional slice tags added

Four nullable fields were added to `ClipRecord` for Issue 004's
slice-based evaluation:

- `lighting` — one of `daylight` / `dim` / `low_light` / `mixed` / `unknown`
- `occlusion` — one of `none` / `partial` / `heavy` / `unknown`
- `multi_person` — bool
- `action_confuser` — one of `none` / `sitting` / `sleeping` / `resting`
  / `exercising` / `crawling` (sourced from PRD user stories 18 / 29)

Each value set is exposed as a frozen constant
(`LIGHTING_VALUES`, `OCCLUSION_VALUES`, `ACTION_CONFUSER_VALUES`) so new
buckets can be added in one line. Unknown values are **warnings**, not
errors — researchers may legitimately need a new bucket, and adding one
should not require a schema bump. Missing tags are not warned about:
placeholder rows are legal until Issue 004 starts aggregating.

The sample manifest demonstrates every tag on real rows so the schema
is documented by example.

### 3. Schema version bumped to 1.1

The loader now checks `schema_version` against `SUPPORTED_SCHEMA_VERSIONS`
and rejects unknown versions loudly. Bumping the manifest without
updating the loader is now a hard error, not a silent fall-through.
Both `1.0` and `1.1` are accepted so old manifests keep loading.

### 4. Dataset staging documentation

`data/README.md` covers the dataset-staging acceptance criterion in full:

- Where each dataset lives on Drive (`datasets/<slug>/`).
- **Store-once rule** — never re-download inside a Colab session.
- **Frozen vault hard wall** — `omnifall`, `caucafall`, `mcfd`, `fallvision`
  may only be staged under their own slugs and only referenced from
  `frozen_unseen_test` manifest rows.
- **Licensed / gated datasets** — manual staging only; no auto-fetch.
- **Manifest paths must be Drive-relative**, not absolute.
- **No raw material in git** — Drive is the data layer, git is the
  source layer.

### 5. Version control readiness

The repo was already initialised as git (per the host environment) but
had zero commits. The initial commit `75c73e8` lands everything in
`.gitignore`'s allow-list:

- `colab/`, `data/`, `tests/`, `requirements.txt`, `context.txt`,
  `issues/` (PRD + per-issue specs as reference).
- All 24 tests pass at commit time.

The `.gitignore` now explicitly excludes Colab-only outputs
(`artifacts/pip_freeze.txt`, `logs/setup_run_log.json`) and raw
dataset material (`*.mp4`, `*.tar`, `datasets/`, `artifacts/crops/`,
`metrics/`, etc.). Those files are produced by the setup notebook
inside a real Colab session on Drive and are intentionally **not**
fabricated in this repo — when they exist they live on Drive, not
locally.

## Important decisions (carried forward from rev 1 + new in rev 2)

1. **Clip-level role locking.** Every clip carries exactly one role.
   Datasets may legitimately span multiple roles (Le2i and GMDCSA-24 are
   documented as cross-listed), but each individual clip is unique and
   single-roled.

2. **Three-axis leakage enforcement.** Clip ID uniqueness + source_path
   disjointness + subject_id disjointness across train and validate.
   Plus the frozen-vault hard wall. These four rules together are the
   only thing standing between honest cross-dataset numbers and a
   misleading result.

3. **Colab-compatible dependency stack.** Single source of truth at
   `colab/setup.py:APPROVED_PIP_PACKAGES`, mirrored in `requirements.txt`.

4. **No reinstall of Colab torch / torchvision.** Post-install sanity
   check (CUDA + cuDNN availability) catches silent breakage.

5. **Frozen unseen-test hard wall.** Vault datasets may ONLY appear in
   `frozen_unseen_test`. The validator catches both directions.

6. **Real `pip freeze` is the env lock.** Written by Colab to
   `artifacts/pip_freeze.txt` on Drive.

7. **TrackEval is not in the default install.** Documented in
   `TRACKEVAL_INSTALL_NOTE`; manual install only.

8. **Schema versioning is enforced, not assumed.** Unknown
   `schema_version` is a hard error so the loader and manifest can't
   silently drift.

9. **Slice tags are advisory, not enforced.** Unknown values warn;
   missing values don't warn. This keeps the validator permissive
   during early data acquisition without letting typos hide.

10. **Dataset staging lives on Drive, source lives in git.** Hard
    rule documented in `data/README.md` and enforced by `.gitignore`.

## Open issues / not yet addressed

- **Subject-level disjointness depends on real subject_id population quality.**
  The check fires whenever two clips in different splits share a
  `subject_id`. But if the manifest author leaves `subject_id=None` for
  most clips, the check has nothing to compare and silently passes. We
  will monitor the real population rate and either: (a) emit a warning
  when a manifest has zero subject_ids on either side of a comparison,
  or (b) require subject_id to be populated for vault-adjacent rows
  before unlocking Issue 004's full evaluation. Decision deferred until
  first real dataset acquisition.
- **Real datasets are not yet acquired.** The `sample_manifest.yaml` is
  placeholder. Real clip IDs, `source_path`s, and provenance will land
  when issue 002/003 begin pulling data.
- **Drive layout has no contents yet.** `MyDrive/fall_detection/` will be
  empty until the notebook is first run in a real Colab session.
- **GPU class is not pinned.** Setup captures whatever GPU Colab hands
  the session. If Colab assigns a smaller GPU (or no GPU at all), the
  downstream pipelines must degrade gracefully.
- **TrackEval install is manual.** No automation; documented in
  `TRACKEVAL_INSTALL_NOTE`.
- **WebDataset shard format not yet chosen.** Issue 003 will pick the
  shard naming / tar format. Setup just keeps the dependency available.
- **No remote git host configured.** Initial commit is local only;
  pushing to GitHub / GitLab is a separate step the project owner can
  take when ready.

## Assumptions

- **Python 3.10+ on Colab.** Colab's current image ships 3.10 / 3.11.
- **PyYAML is available.** It is in the approved stack and required by
  the manifest loader. The loader surfaces a clear ImportError if missing.
- **Drive mount succeeds.** If running outside Colab, `DriveLayout`
  falls back to a local path via `FALL_DETECTION_DRIVE_ROOT`.
- **`nvidia-smi` is on PATH inside the Colab runtime.** If not, the
  run log records `gpu_name=null` rather than crashing.
- **The dataset slugs in `IN_SCOPE_DATASETS` / `FROZEN_VAULT_DATASETS`
  match the PRD.** Confirmed against `issues/prd.md` Phase I decisions.
- **The `_DASH` convention for dataset slugs is stable.** Source files
  use lowercase + underscores (`up_fall`, `caucafall`); the loader
  normalises to lowercase so `URFD`, `urfd`, `UrFd` all parse identically.
- **Local git commits only.** No remote is configured; `git push` will
  fail until the owner wires one up. This is intentional and not a bug.
